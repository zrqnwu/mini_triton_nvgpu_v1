## 一、先定学习标准：什么叫“学会了”

这份学习路径不追求“把领域一次学完”，而追求“学到能把当前项目补起来”。

建议把学习深度分成 4 档：

### L0：知道名词

- 能大概知道它是什么
- 但还不能用它指导代码修改

### L1：能读懂当前项目

- 能看懂当前仓库里相关结构和 pass 在做什么
- 能知道哪里是 owner，哪里只是消费方

### L2：能独立修改当前项目

- 能自己判断应该改哪层
- 能补字段、改 pass、补校验，而不是只会跟着现有代码抄

### L3：能解释 tradeoff

- 能解释为什么这么设计
- 能说明不这么做会出什么问题
- 能把当前实现和 Triton 思想联系起来

对你现在这个项目来说，绝大多数学习项的目标都不是 L3 全覆盖，而是：

- stage1 主线相关内容至少做到 `L2`
- 关键架构边界内容尽量做到 `L3`

## 三、第一阶段：补足当前主线时必须学会的内容

## 1. MLIR 里 attr / type / op / pass 的职责边界

### 要学的具体知识

- `attr` 适合表达什么：
  - 静态合同
  - 编码、布局、mapping、target truth
- `type` 适合表达什么：
  - 值本身的语义和承载约束
- `op` 适合表达什么：
  - 显式执行边界
  - 不能只靠 attr 讲清的行为
- `pass` 适合表达什么：
  - 分析
  - 重写
  - cleanup
  - legality / verify
  - lowering
- `analysis contract` 和 `IR owner` 的区别
- “谁做决策，谁只消费”的边界

### 学到什么程度算够

- 能分清：
  - 什么应该放 `EncodingPlan / ProgramMappingPlan / EpiloguePlan`
  - 什么应该放 `tb.pipeline_mainline`
  - 什么不应该拖到 `LowerPipelineToNVGPU`
- 看到一个新需求时，能先判断“该改哪层”，而不是直接去 lowering 里加分支

### 暂时不用深挖

- MLIR 全部 dialect conversion 细枝末节
- region/trait/interface 的全量体系
- 太通用的 compiler theory

### 学完自测

你至少要能回答这 4 个问题：

1. 为什么 `shared swizzle` 应该是 encoding owner，而不是 lowering 私有推导？
2. 为什么 `pipeline_mainline` 需要是显式 op，而不是 lowering 临时重建？
3. 为什么 `target lowering` 最好只做翻译，不做上层决策？
4. 为什么 verifier pass 不应该等价于 lowering 里的 if/else 检查？

### 回到项目里该先看什么

- `include/tb/Analysis/*.h`
- `include/tb/IR/TBOps.td`
- `include/tb/Transforms/Passes.td`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`

目标深度：

- `L3`

## 2. Triton matmul 的真实编译顺序

### 要学的具体知识

- Triton 风格主线不是“先做 kernel 技巧”，而是先定语义真相
- 编译顺序大致是：
  - matmul 语义化
  - program mapping / layout
  - matmul 专项 rewrite
  - loop regularization
  - latency / schedule / waits
  - pipeline expansion / cleanup
  - target lowering
- 为什么 layout / CTA / operand fragment 要尽早固定
- 为什么 pipeline 和 lowering 不应该再回头猜 fragment/layout

### 学到什么程度算够

- 能把你当前项目的主线 pass 顺序讲清楚
- 能指出当前哪些地方已经对齐 Triton 思想，哪些地方只是“结构有了但覆盖还没补完”

### 暂时不用深挖

- Triton 全部 pass 的实现细节
- Triton 所有 backend 的差异

### 学完自测

你要能不用看稿，讲出下面这段逻辑：

- 为什么 `ProgramMappingPlan` 比 `LowerPipelineToNVGPU` 更应该拥有 mapping 真相
- 为什么 `EpiloguePlan / TargetLandingPlan` 应该在 lowering 前决定好 C landing
- 为什么 general-shape 没收口时，不应该靠 lowering 末端硬补

### 回到项目里该先看什么

- `docs/stage1_data_structures_and_passes.md`
- `lib/Transforms/Stage1Pipelines.cpp`
- `lib/Analysis/ProgramMappingPlan.cpp`
- `lib/Analysis/EncodingPlan.cpp`
- `lib/Analysis/EpiloguePlan.cpp`

目标深度：

- `L3`

## 3. Ampere 上的 `cp.async / ldmatrix / mma.sync`

### 要学的具体知识

- `cp.async` 的作用和基本约束：
  - global -> shared
  - 字节粒度
  - async group / wait 关系
- `ldmatrix` 的作用：
  - shared -> register fragment
  - 为什么 shared 物理布局必须配合它
- `mma.sync m16n8k16` 的基本 fragment 组织
- warp 内 fragment 和 lane 的关系
- 为什么 A/B shared tile 不是随便摆都行

### 学到什么程度算够

- 能解释当前 exact-tile 主线为什么快
- 能知道 shared 布局一改，为什么 `ldmatrix` / fragment / bank conflict 也要一起重看

### 暂时不用深挖

- Hopper/Blackwell 新指令

### 学完自测

至少能回答：

1. 为什么 `cp.async` 不只是“比 load/store 快”，而是 pipeline overlap 的一部分？
2. 为什么 `ldmatrix` 对 shared 物理布局敏感？
3. 为什么 `mma.sync` 的 fragment 组织会反过来约束 layout plan？

### 回到项目里该先看什么

## 五、每一块学完以后，怎么证明自己不是“看懂了但不会用”

每学完一个主题，都建议做三件事：

### 1. 用自己的话复述一遍

至少要能不用照稿讲清：

- 这个东西解决什么问题
- 为什么在当前项目里重要
- 它应该放在哪一层

### 2. 回到仓库里定位 2 到 4 个相关文件

不要停在概念层。  
你必须能把抽象知识映射到当前项目的真实代码位置。

### 3. 做一个最小改动或最小测试

比如：

- 加一个 verifier 检查
- 加一个负例测试
- 给一个 attr 补一个字段
- 给 benchmark 补一个 shape

如果完全没有落到仓库里，这块知识大概率还停留在 `L0/L1`。

- 能转化成 pass 边界判断

- 能转化成 benchmark 与性能归因能力
- 能直接支撑你把当前项目补到更成熟的工程状态
