## 二、当前真正没完成的地方

按优先级看，剩余未完成项分成两层。

### 2.1 Stage1 当前必须完成的

#### 1. multi-warp general-shape 的 C epilogue 还没有 Triton 等价

现状：

- 当前 `160x96x32_general` 仍明显慢于 Triton
- 当前代码里已经把 `shared-pack` 的 scratch 从更大的形态压到更小，但仍然不是 Triton 那种 shared owner 组织
- 当前 `lib/Analysis/EpiloguePlan.cpp` 中的 `shared_pack_slots` 规则仍是阶段性策略，不是最终结构真相

直接证据：

- 当前实现位置：
  - `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/EpiloguePlan.cpp`
- 当前规则位置：
  - `shared_pack_slots = config.numWarps == 1 ? packsPerWarpRow : 1`
- Triton 对照 PTX：
  - `/tmp/triton_160x96x32_general_inspect/kernel.ptx`
- Triton metadata：
  - `/tmp/triton_160x96x32_general_inspect/kernel.json`

结论：

- 这不是主循环 A/B async 的问题
- 这是 C epilogue 重排 owner 还没按 Triton 方式建模

#### 2. shared memory owner 仍不是 Triton 式统一 CTA workspace

现状：

- 当前 lowering 仍为 A、B、C 分别准备 workgroup attribution
- exact-tile 时问题不大
- multi-warp general-shape 时，这会让 C reorder scratch 很难与 mainloop shared 做真正 alias
- 当前 `LowerPipelineToNVGPU.cpp` 仍以“多个独立 workgroup memref”的方式组织 shared

这和 Triton 的关键差别是：

- Triton 最终 PTX 使用的是单个 `.extern .shared .b8 global_smem[]`
- 也就是一个统一 CTA shared workspace
- A/B staging、epilogue reorder、其他 scratch 都是在同一块 shared 空间里切片与复用

直接证据：

- Triton PTX 顶部：
  - `/tmp/triton_160x96x32_general_inspect/kernel.ptx`
- 当前 mini lowering：
  - `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/LowerPipelineToNVGPU.cpp`

结论：

- 当前 shared owner 的根问题不是“shared 太多”这么简单
- 而是 shared 的组织方式还不是 Triton 的“统一 workspace + alias by lifetime”

#### 3. `TargetLandingPlan` 仍然把三件事混在一起

当前 `TargetLandingPlan` 同时在表达：

- 最终 landing kind
- 是否需要 shared reorder
- shared reorder scratch 的具体分配大小

这三件事在 Triton 思想里不是一个层次：

- landing kind 决定“最终回 global 的行为”
- reorder owner 决定“fragment 如何重排成 global row vectors”
- workspace owner 决定“这段 reorder 用哪块 shared、何时活着、是否能 alias”

现在把它们揉成一个 plan，会导致：

- EpiloguePlan 做太多
- ResourceClosurePlan 无法做真正 fixed-point
- Lowering 末端还得补额外判断

结论：

- 必须把 `landing`、`reorder`、`workspace` 三层拆开

#### 4. `ResourceClosurePlan` 现在仍然是“相加式预算”，不是 Triton 式 peak shared fixed-point

现状：

- 当前 closure 仍主要输出：
  - `estimated_ab_shared`
  - `estimated_epilogue_shared`
  - `estimated_total_static_shared`
- 这个 `total` 还是以“相加”为主

但 Triton 真正需要的是：

- shared 不是按语义角色相加
- 而是按 lifetime 求 peak
- A/B mainloop shared 与 epilogue reorder scratch 若不重叠，就必须允许 alias

这就是当前 `160x96x32_general` 没法进一步贴近 Triton 的核心原因之一。

#### 5. `WarpDecompositionPlan` 仍没有显式表达 epilogue reorder mapping

当前 `WarpDecompositionPlan` 里有：

- `accumulatorPackIds`
- `epiloguePackIds`
- `sharedPackIds`

但没有真正表达：

- fragment 到 shared reorder tile 的 row/col 映射
- row-vector producer 与 consumer 的关系
- 哪些 row vectors 必须跨 fragment 组合
- 哪些重排是 lane-local，哪些必须借助 shared

这会导致：

- direct path 和 shared-pack path 的边界只能靠启发式判断
- 一旦 shape 复杂，plan 只能退回经验规则

#### 6. 当前 multi-warp general-shape 的硬件闭环还没有达到 Triton

这个不再是“代码路径没覆盖”，而是：

- 代码路径存在
- correctness 也已经通
- 但性能仍没达到 Triton

所以这条不能再用“以后再看”带过去，必须当作 stage1 尚未完成。

## 三、当前一级根因

如果只看“现在为什么还没完全达到 Triton”，一级根因只有一个：

### 当前 shared owner 还是按语义角色分裂的，不是按 Triton 的统一 CTA workspace + lifetime alias 来建模

它具体表现成四件事：

1. C epilogue reorder scratch 仍是单独思维，不是 CTA shared workspace 的一个 interval
2. `TargetLandingPlan` 把 landing/reorder/workspace 混成一个 plan
3. `ResourceClosurePlan` 不能按 shared live range 做 peak closure
4. `LowerPipelineToNVGPU` 仍按多个独立 workgroup attribution 组织 shared

这四件事叠在一起，结果就是：

- exact-tile 不容易出问题
- 单 warp general-shape 也还能顶住
- 但 multi-warp general-shape 会把 shared 成本放大
- 放大后就和 Triton 拉开差距

---

## 四、为什么“直接把 shared-pack 拔掉”不是正确方案

这一点必须明确写出来，避免后面再走错。

这轮已经证明：

- 直接把 multi-fragment direct pack 强行改成 pure register assemble/disassemble
- correctness 会立刻出错

根因不是“代码没写对”那么简单，而是：

- 当前这些 multi-fragment direct packs 的列重排不是纯 lane-local
- shared bridge 在这里不是纯粹多余的 staging
- 它承担了真实的 reorder 职责

这和 Triton PTX 也是一致的：

- Triton 在 `160x96x32_general` 上也确实有
  - `st.shared.v2.b32`
  - `ld.shared.v4.u32`
  - `bar.sync`

所以严格对齐 Triton 的正确方向不是：

- “把 shared reorder 完全删掉”

而是：

- “把 shared reorder 的 owner 改成 Triton 式统一 CTA workspace 下的紧凑 scratch / alias scratch”

## 5.1 新增 `SharedWorkspacePlan`，把 shared 真相从“按角色分配”改成“按生命周期分配”

这是这次最核心的新增结构。

### 新增文件

- `include/tb/Analysis/SharedWorkspacePlan.h`
- `lib/Analysis/SharedWorkspacePlan.cpp`

### 修改文件

- `include/tb/Analysis/KernelContract.h`
- `lib/Analysis/KernelContract.cpp`
- `lib/Transforms/BuildProgramMappingPlan.cpp`
- `lib/Transforms/Stage1Pipelines.cpp`
- `lib/Transforms/CleanupPipeline.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`

### 要表达的真相

`SharedWorkspacePlan` 必须显式表达：

- CTA shared 总空间大小
- workspace 中每个 segment 的：
  - 逻辑角色
  - byte offset
  - byte size
  - alignment
  - producer
  - consumer
  - lifetime begin
  - lifetime end
  - 是否允许 alias
  - alias set
- peak shared bytes

### 至少要有的 segment 类型

- `mainloop_a_stage_buffer`
- `mainloop_b_stage_buffer`
- `epilogue_reorder_scratch`
- `splitk_reduction_scratch`
- `persistent_state_scratch`

### 核心原则

1. shared owner 必须统一成一个 CTA workspace
2. A/B/C/reduction/persistent 都只是 workspace segment，不再各自拥有独立 shared backing 真相
3. 是否能 alias，只由 lifetime 和 alignment 决定，不由语义角色决定

### 为什么这一步必须做

因为 Triton 的真实 shared owner 就是单个 `global_smem`。

只要你还保持：

- A 一块 shared
- B 一块 shared
- C 一块 shared

这种分裂式 owner，multi-warp general-shape 就不可能真正对齐 Triton。

## 5.2 新增 `EpilogueReorderPlan`，把 landing / reorder / workspace 三层拆开

### 新增文件

- `include/tb/Analysis/EpilogueReorderPlan.h`
- `lib/Analysis/EpilogueReorderPlan.cpp`

### 修改文件

- `include/tb/Analysis/EpiloguePlan.h`
- `lib/Analysis/EpiloguePlan.cpp`
- `include/tb/Analysis/KernelContract.h`
- `lib/Analysis/KernelContract.cpp`
- `lib/Transforms/CleanupPipeline.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`

### 当前必须拆开的三层

#### 层 1：`TargetLandingPlan`

只负责：

- 最终 global vector landing
- vector width
- boundary aware / masked global IO

不再负责：

- reorder scratch 大小
- 是否 alias
- scratch slot 数

#### 层 2：`EpilogueReorderPlan`

负责：

- 是否需要 reorder
- reorder kind
  - `none`
  - `cta_shared_row_reorder`
  - `cta_shared_relay`
- fragment -> row vector 的 mapping
- shared row vector store/load width
- required barrier kind
- required barrier count
- live slots

#### 层 3：`SharedWorkspacePlan`

负责：

- reorder scratch 放在哪块 shared
- 是独占 segment 还是 alias segment
- 实际 byte offset / size / alignment

### 为什么必须拆

因为 Triton 思想里：

- “最终怎么回 global”
- “中间是否需要 reorder”
- “reorder 用哪块 shared”

本来就是三层不同 owner。

当前把它们揉在 `TargetLandingPlan` 里，是当前结构不彻底的根源。

## 5.3 重写 `ResourceClosurePlan`，从“相加预算”改成“shared peak fixed-point”

### 修改文件

- `include/tb/Analysis/ResourceClosurePlan.h`
- `lib/Analysis/ResourceClosurePlan.cpp`
- `lib/Transforms/CleanupPipeline.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`

### 新的输入

必须显式消费：

- `ProgramMappingPlan`
- `WarpDecompositionPlan`
- `EpiloguePlan`
- `EpilogueReorderPlan`
- `SharedWorkspacePlan`
- `ReductionPlan`
- `PersistentWorkPlan`
- `EncodingPlan`
- `TransportPlan`

### 新的输出

除了当前已有字段，还必须补：

- `peak_static_shared_bytes`
- `workspace_total_bytes`
- `workspace_alias_saved_bytes`
- `reorder_shared_bytes`
- `reduction_shared_bytes`
- `persistent_shared_bytes`
- `mainloop_shared_live_begin/end`
- `epilogue_reorder_live_begin/end`
- `selected_shared_workspace_policy`

### 关键算法

`ResourceClosurePlan` 必须做的是：

1. 先算每类 scratch 的真实 byte 需求
2. 再根据 lifetime 建 interval
3. 对 interval 做 alias packing
4. 输出 peak shared，而不是简单求和

这一步必须成为唯一 shared 真相，不能再让 lowering 自己额外猜。

### 当前必须删掉的错误思路

- `estimated_total_static_shared = estimated_ab_shared + estimated_epilogue_shared`

这只能作为过渡期近似，不能作为最终结构。

## 5.4 扩 `WarpDecompositionPlan`，把 epilogue reorder mapping 做成第一类真相

### 修改文件

- `include/tb/Analysis/WarpDecompositionPlan.h`
- `lib/Analysis/WarpDecompositionPlan.cpp`
- `lib/Analysis/EpilogueReorderPlan.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`

### 必须补的字段

对每个 warp，除了现有的：

- `accumulatorPackIds`
- `epiloguePackIds`

还要补：

- `reorderRowVectorIds`
- `fragmentToRowVectorMap`
- `rowVectorStoreOrder`
- `rowVectorLoadOrder`
- `reorderNeedsShared`
- `reorderTileBase`

### 作用

这一步的目标是：

- direct path 是否成立，不再靠启发式
- shared reorder 的最小工作集，不再靠经验槽数
- reorder scratch 的 live slots，从 mapping 真相推导，而不是硬编码

### Triton 对齐原则

Triton 不会在最后阶段再“临时猜哪些 fragment 可以直接拼”。  
它前面已经把 layout / fragment / warp decomposition 的真相冻结好了。

所以这里必须把 `epilogue reorder mapping` 前移成显式 plan。

## 5.5 重写 `LowerPipelineToNVGPU` 的 shared 组织：从多个 workgroup attribution 改成单个 CTA workspace

这是最终性能是否真能对齐 Triton 的关键。

### 修改文件

- `lib/Transforms/LowerPipelineToNVGPU.cpp`

### 当前必须拔掉的旧结构

当前 lowering 里这套思路必须退役：

- workgroup attribution[0] = A shared
- workgroup attribution[1] = B shared
- workgroup attribution[2] = C shared pack

### 新结构

改成：

- 一个统一 `workgroup` shared workspace attribution
- 类型建议：
  - `memref<peak_bytes x i8, 3>` 或等价字节视图
- 然后由 typed `view/reinterpret_cast/subview` 派生：
  - `sharedA`
  - `sharedB`
  - `epilogueReorderScratch`
  - 后续 `splitkScratch`
  - 后续 `persistentScratch`

### 为什么这一步是 Triton 思想

因为 Triton 的 PTX 本质就是：

- 一个 `global_smem`
- 上层所有 shared owner 都在里面切片

如果你这里还保持多个独立 attribution，就只是“语义上像”，不是“owner 上像”。

### C reorder 的 lowering 规则

#### 1. exact-tile

- 不走 reorder scratch
- 直接 register pack -> global vector IO

#### 2. single-warp general-shape

- 若 mapping 要求 shared reorder，则使用 workspace 中的 reorder segment
- 但允许保留整批 row vectors，减少 barrier

#### 3. multi-warp general-shape

- 必须使用 alias 到 CTA workspace 的紧凑 reorder scratch
- 只保留最小 live slots
- 不允许再按当前“独立 C workgroup attribution”思路单独膨胀 shared

### 这里必须坚持的原则

1. Lowering 不再决定“是否 alias”
2. Lowering 只消费 `SharedWorkspacePlan`
3. Lowering 不再凭经验决定 C scratch 的大小

## 八、这份方案的核心原则

最后把最重要的原则再钉一次。

### 原则 1

不要再把“shared reorder 是否存在”当成问题。  
真正的问题是“它由谁拥有、怎么分配、是否能 alias”。

### 原则 2

不要再尝试把所有 multi-fragment general-shape C path 都改成纯 register direct。  
这不符合当前事实，也不符合 Triton PTX。

### 原则 3

最终要对齐 Triton，不是“看起来像 Triton 的分支命名”，而是：

- shared owner 组织像 Triton
- resource closure 像 Triton
- epilogue reorder owner 像 Triton
- 最终硬件结果也像 Triton

### 原则 4

所有未来新增主线：

- split-k
- persistent
- 更复杂 general-shape

都必须建立在统一 `SharedWorkspacePlan` 之上。  
如果继续给每条主线单独开一份 shared owner，以后还会重复今天的问题。

---

## 九、当前一句话结论

当前剩余未完成项的真正根因，不是某个参数没调好，而是：

**shared owner 还没有按 Triton 的统一 CTA workspace + lifetime alias 真相重构完。**

只要把这一层彻底做对：

- 当前 `160x96x32_general` 这类 multi-warp general-shape 才有机会真正贴近 Triton
- 后续 split-k / persistent 也才有统一、稳定、不会再反复返工的地基
