## 一、先说结论

如果你现在只有基础 `MLIR` 和基础 `CUDA`，这个项目是可以开始读的。

但正确读法不是：

- 一上来先啃最大文件
- 先看 `LowerPipelineToNVGPU.cpp`
- 先试图把所有 `.cpp` 逐行看懂

正确读法是：

1. 先建立整条主线
2. 再看每一层谁拥有真相
3. 再看这些真相为什么必须按当前顺序被建立
4. 最后才去看具体 lowering

对这个项目来说，最重要的不是先记住实现细节，而是先建立下面这套认知：

- 这个项目试图把哪些“硬真相”提前显式化
- 每一层各自负责什么
- 为什么后一层不能重新猜前一层
- 为什么 pass 顺序必须按依赖链来

## 二、读这个项目之前，你至少要具备哪些基础知识

这里不是让你先把整门领域学完，而是列出“足够支撑你开始读代码”的最低基础。

## 1. MLIR 最低基础

你至少要理解：

- `op` 是什么
- `attr` 是什么
- `type` 是什么
- `pass` 是什么
- pass 可以做哪些事：
  - verification
  - analysis
  - rewrite
  - cleanup
  - lowering

对这个项目来说，最关键的一点不是 API，而是：

- 你要能区分“谁拥有真相”
- 和“谁只是消费真相”

你至少要能看懂下面这些概念：

- 高层 op 只拥有语义，不拥有全部物理细节
- 一部分真相放 attr/type
- 一部分真相放 analysis contract
- 一部分执行边界必须做成显式 op
- lowering 理想情况下只做翻译，不重新做上层决策

## 2. CUDA / GPU 最低基础

你至少要理解：

- `grid / block / thread / warp / lane`
- `global memory / shared memory / register`
- 为什么 shared memory 比 global 更快
- 为什么 register 和 shared 使用量会影响 occupancy
- Tensor Core 主线里这些词在干什么：
  - `cp.async`
  - `ldmatrix`
  - `mma.sync`

你不用一开始就把指令手册背下来，但至少要知道：

- `cp.async` 是 global -> shared 的异步搬运
- `ldmatrix` 是 shared -> register fragment
- `mma.sync` 是 Tensor Core 的矩阵乘运算

## 3. 矩阵乘法分层最低基础

你至少要理解：

- `problem shape`
- `CTA tile`
- `warp tile`
- `instruction tile`

如果这个分层没有概念，你会看不懂：

- `ProgramMappingPlan`
- `EncodingPlan`
- `AccumulatorPlan`

## 4. 性能工程最低基础

你至少要理解：

- 什么叫公平 benchmark
- 什么叫 warmup
- 什么叫 `register pressure`
- 什么叫 `shared memory pressure`
- 什么叫 `occupancy`

这个项目不是只为了“能编译”，而是为了“能按 Triton 思想走出正确主线并打出性能”。

## 5. 现在还不用先学到很深的东西

你现在不用先深挖：

- LLVM backend 细枝末节

## 四、为什么这个项目会这样设计

这部分是你读项目时最该先理解的东西。

## 1. 为什么 `tb.matmul` 被设计得比较“瘦”

看 [TBOps.td](/home/zhangruiqi/mini_triton_nvgpu_v1/include/tb/IR/TBOps.td#L32)，`tb.matmul` 现在只拥有：

- 输入 memref
- block tile
- `group_m`
- `exact_tile`

它没有直接拥有：

- shared layout
- dot fragment
- transport 策略
- epilogue landing
- pipeline 主线

为什么这样设计？

因为高层 op 应该先只表达：

- 问题语义
- tile 语义

而不是一开始把所有物理真相都写死。

如果把所有物理细节都塞进高层 op，后面 pass 的分层就会变糊，项目会退化成：

- 一个看似高层、实则夹杂大量后端假设的巨型 op

## 2. 为什么要把 `ProgramMappingPlan / EncodingPlan / TransportPlan / EpiloguePlan` 拆开

因为这些不是同一类问题。

- `ProgramMappingPlan`
  回答“哪个 program / CTA 负责哪块 tile”
- `EncodingPlan`
  回答“值在物理上怎么摆”
- `TransportPlan`
  回答“global 到 shared 怎么搬”
- `EpiloguePlan`
  回答“C 最后怎么落地”

如果把这些全塞进一个 plan 或一个 lowering 文件里，前期可能快，但后期一定会出现：

- owner 混乱
- 责任边界不清
- lowering 末端开始重建上层真相

## 3. 为什么 `tb.pipeline_mainline` 必须做成显式 op

看 [TBOps.td](/home/zhangruiqi/mini_triton_nvgpu_v1/include/tb/IR/TBOps.td#L60)，`tb.pipeline_mainline` 是 cleanup 后、lowering 前的显式执行 owner。

它的意义是：

- 到了这一层，执行主线已经必须公开成显式 IR
- target lowering 不应该再临时重建 pipeline skeleton

为什么这一步重要？

因为如果 `pipeline_mainline` 不显式化，后面的 target lowering 很容易演化成：

- 一边 lowering
- 一边重新猜 stage / cluster / wait / producer-consumer 关系

这正是这个项目想避免的事情。

## 4. 为什么要把一部分 verifier 前移

现在项目里已经补了 `tb-verify-pipeline-mainline-contract`，见 [Passes.td](/home/zhangruiqi/mini_triton_nvgpu_v1/include/tb/Transforms/Passes.td#L184)。

这背后的设计理由是：

- 错误越早暴露越好
- verifier 应该尽量在 lowering 前把 contract 问题说清楚
- lowering 最好少背“合同已经坏了”的锅

这和真实编译器工程非常一致。

## 六、为什么 pass 顺序必须是这个顺序

主线顺序定义就在 [Stage1Pipelines.cpp](/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/Stage1Pipelines.cpp#L23)。

这里不是“习惯排序”，而是“依赖排序”。

## 1. `tb-verify-scope`

先确认当前 op 落在合法范围内。

为什么必须最先做：

- 如果 case 本身超出 `stage1` 支持范围
- 后面所有合同和 lowering 分析都没有意义

## 2. `tb-attach-target-info`

把硬件真相挂出来。

为什么必须很早做：

- layout
- transport
- Tensor Core 指令选择
- lowering

全都依赖 target 能力。

## 3. `tb-semanticize-matmul`

把高层 op 变成更明确的 matmul 语义。

为什么在 layout 之前：

- 先要知道“这是一个什么 matmul”
- 才能谈“值在物理上怎么摆”

## 4. `tb-build-program-mapping-plan`

先定 CTA 外部映射。

为什么在 layout/pipeline 之前：

- 先知道哪个 CTA 负责哪块 tile
- 才能继续讲 CTA 内部主线怎么安排

## 5. `tb-build-layout-plan`

先定编码和 fragment 真相。

为什么在 transport 之前：

- 先知道 shared/dot/acc 的物理形态
- 才能合法决定一次搬多少、能不能 async

## 6. `tb-build-transport-plan`

再定搬运策略。

为什么不和 layout 混在一起：

- layout 回答“怎么摆”
- transport 回答“怎么搬”

它们相关，但不是同一类 owner。

## 7. `tb-build-c-register-plan`

决定 accumulator 和 epilogue 的基本合同。

为什么在 rewrite 之前：

- rewrite 时已经要知道 C path 是否能 direct
- 不能等 lowering 才突然决定 C 怎么落地

## 8. `tb-rewrite-matmul-mainloop`

把前面的合同收束成 tensor-core mainloop 形态。

为什么现在做：

- 语义、mapping、layout、transport、C path 都已经决定得差不多了
- 现在才具备做 mainloop 专项重写的条件

## 9. `tb-cleanup-layout-conversions`

收口显式 layout conversion。

为什么在 mainloop rewrite 之后：

- 先让 layout truth 和 rewrite 结构显式出来
- 再清理冗余的 `tb.convert_layout`

## 10. `tb-build-mainloop-graph`

构建统一资源图，也就是 `BufferModel`。

为什么在 rewrite 之后：

- 资源图应该建立在“真正 mainloop 结构”之上
- 不能过早建立，否则会混进太多前期结构噪音

## 11. `tb-regularize-k-loop`

把主线规整成单一 K-loop owner。

为什么在 latency/schedule 之前：

- 没有稳定 loop 形态
- 后面的 latency 和 schedule 就没有统一对象

## 12. `tb-assign-latencies`

给 op 标显式延迟合同。

为什么在 schedule 之前：

- schedule 需要知道 op 延迟
- 不能在 schedule 末端现猜

## 13. `tb-schedule-loops`

决定 stage / cluster / order。

为什么在 derive waits 之前：

- 没有 schedule
- 就没有 wait frontier 的锚点

## 14. `tb-derive-waits`

根据 async producer 和 schedule 推 wait。

为什么现在做：

- 这时 producer、consumer、stage 都已经有了
- 等待关系终于可以精确落下来

## 15. `tb-expand-pipeline`

把抽象 pipeline 真正展开成 cluster 序列。

为什么在 cleanup 之前：

- cleanup 要消费的是“明确展开后的主线”
- 不是抽象 schedule

## 16. `tb-cleanup-pipeline`

把 `pipeline_expansion` 收口成 `tb.pipeline_mainline`。

为什么这一层重要：

- 到这里以后，主线执行真相必须已经显式公开
- 后端不应该再私下重建

## 17. `tb-verify-pipeline-mainline-contract`

在 lowering 前检查 mainline contract。

为什么放在这里：

- 这是 lowering 前最后一层 verifier
- 让 target lowering 只做翻译，不承担太多“晚发现合同错误”的职责

## 18. `tb-lower-pipeline-to-nvgpu`

把公开的合同翻译到 GPU/NVGPU。

为什么不能更早做：

- 因为只有现在所有上层 owner 都收口了
- lowering 才能真正“只消费，不重建”

## 19. `tb-lower-epilogue-vector-io`

把 late direct-global vector IO 降成稳定 memref vector 访问。

为什么放在 NVGPU -> NVVM sink 前：

- 因为如果太早走 generic 路径
- direct-global vector IO 很容易被提前 scalarize

这也是 [Stage1Pipelines.cpp](/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/Stage1Pipelines.cpp#L48) 里中文注释强调的地方。

## 九、读项目时最常见的误区

## 1. 一上来先啃最大文件

这会让你只看到结果，看不到为什么。

## 2. 只看 `.cpp`，不先看 `.h` 和 `Passes.td`

这样很容易把“实现细节”误当成“设计边界”。

## 3. 只按功能模块读，不按 pass 依赖读

编译器项目最怕这个。  
这个项目的核心逻辑不是文件夹分类，而是“谁依赖谁”。

## 4. 只关心能不能 lower，不关心 owner truth

这样会把项目看成普通 codegen demo，看不到它真正值钱的地方。

## 5. 过早去纠结 stage2

你现在最重要的是先把当前 stage1 主线读顺。  
`split-k / persistent / autotune` 不是第一周的重点。

## 十、一句话总结

如果你现在只有基础 `MLIR + CUDA`，读这个项目最正确的方式不是“先把所有底层都学完”，而是：

1. 先看主线 pass 顺序
2. 再看每层 owner truth
3. 再理解为什么必须按这个顺序建立这些真相
