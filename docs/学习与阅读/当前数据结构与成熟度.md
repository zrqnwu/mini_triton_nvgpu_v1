# 当前数据结构与成熟度

更新时间：2026-04-06

适用对象：`/home/zhangruiqi/mini_triton_nvgpu_v1`

本文档只回答两件事：

1. 当前哪些数据结构和 pass 已经成熟到可以当长期地基
2. 当前哪些部分还只是阶段性可用，后续还需要继续扩

这里的“成熟”不是指“字段已经存在”，而是指：

- 已经进入当前 `stage1 exact-tile matmul` 的真实主线
- 后续 pass / lowering 已经实际消费
- 端到端已经形成可编译、可运行、可测的闭环

---

## 一、当前性能结论

基于最近一次公平 benchmark 结果，当前框架在 `stage1 + single-CTA + exact-tile` 这个范围内，性能已经是强的。

最近一轮公平测试结论：

- GPU 预热后从 `P8 / 210MHz` 拉到 `P0 / 1980MHz`
- 6 个 exact-tile shape 上，mini 当前都快于对比用 Triton cubin
- 代表性结果：
  - `64x128x32`: mini `3091.92 ns`，Triton `4031.28 ns`
  - `128x128x32`: mini `4554.96 ns`，Triton `5889.43 ns`
  - `128x128x64`: mini `6356.18 ns`，Triton `7705.14 ns`

这说明当前主线不是“概念正确但实现还很慢”，而是：

- 主线结构已经成立
- 真实 lowering 和执行形态已经能打到很有竞争力的性能

但这个结论有边界：

- 当前最强的是 `single-CTA`
- 当前最强的是 `exact-tile`
- 当前最强的是 `direct-global-vector epilogue`
- 不能把这组结果直接外推成“已经完成全量 Triton matmul 覆盖”

---

## 二、数据结构成熟度分级

### A级：已经可以当长期地基

这些结构我认为已经不用推倒重来。后续即使继续扩 general-shape、shared-relay、split-k/persistent，也应该在这些结构之上继续加能力，而不是换一套新骨架。

#### 1. 入口上下文层

- `KernelConfig`
- `TargetInfo`
- `MatmulSemantics`

成熟原因：

- 已经把问题规模、tile 请求、`numWarps`、`requestedStages`、硬件能力、exact-tile 语义拆清楚
- 整条 stage1 pass 链都在消费这几层信息
- 没有再回退到 lowering 末端临时猜参数

结论：

- 这三层已经可以固定下来

#### 2. 布局与运输主合同

- `EncodingPlan`
- `TransportPlan`

成熟原因：

- `EncodingPlan` 已经把 blocked/shared/dot/accumulator 编码和 operand fragment lane 访问公开化
- `ldmatrix` 需要的 tile count / transpose / register shape 已经在合同中
- `TransportPlan` 已经把 async/vector/cache legality 从 layout 层独立出来
- lowering 已经真实消费这些合同，不是摆设

结论：

- 结构本身已经成熟
- 后续主要是“补更强策略”，不是换结构

#### 3. C 寄存器侧主合同

- `AccumulatorPlan`

成熟原因：

- accumulator pack、lane access、owner scope 都已经稳定
- epilogue 和 lowering 都真实依赖它
- 这是当前 direct-global-vector path 的关键地基

结论：

- `AccumulatorPlan` 已经可以作为长期稳定层

#### 4. 主线资源图与流水层

- `BufferModel`
- `LoopPlan`
- `LatencyPlan`
- `PipelinePlan`
- `AsyncPlan`
- `PipelineExpansion`
- `PipelineReady`

成熟原因：

- 这几层已经构成当前最成熟的一条闭环
- 它们已经把资源图、K-loop、latency、schedule、wait frontier、expanded clusters 串起来
- 后面还有显式 `tb.pipeline_mainline` 作为 lowering owner

结论：

- 这是当前仓库里最值钱、最不该再推倒的一组

#### 5. lowering 入口总合同

- `KernelContract`
- `tb.pipeline_mainline`

成熟原因：

- `KernelContract` 已经把 lowering 真正依赖的合同集中收口
- `tb.pipeline_mainline` 已经把 cluster 序列公开成 IR，而不是留给后端 executor 黑箱重建

结论：

- 这是你当前最接近 Triton 思想的地方之一
- 应该保住，不要再退回“side-car plan + lowering 再猜一次”

### B级：结构方向对，但覆盖范围还不完整

这些结构不是“有问题需要推翻”，而是：

- 当前 exact-tile 主线里已经部分成立
- 但还没有扩到你后续真正想要的完整 matmul 范围

#### 1. `ProgramMappingPlan`

当前状态：

- 对 `tile / grouped_tile` 已经可用
- 对 `split-k / persistent` 还只是计划层表达
- `grouped_tile` 已经进入 strict lowering 与回归测试
- `split-k / persistent` 仍未进入执行主线

为什么是 B 级：

- 上层字段和枚举已经有了
- 但 lowering 还明确拒绝 `split-k / persistent`
- 所以它现在是“结构对了，但执行覆盖还没跟上”

结论：

- 不需要换结构
- 需要补完整 reduction scratch / persistent tile fetch 这类 owner

#### 2. `EpiloguePlan`

当前状态：

- 对 `direct-global-vector` 主线已经可用
- `boundary-aware direct` 和 `shared-pack` 已经进入 cleanup 与 lowering 主线
- 对 `shared-relay / general-shape` 的完整 runtime/perf 闭环还不完整

为什么是 B 级：

- 类型设计已经留了 direct 与 relay 的分叉
- 当前 exact-tile 主线已经稳定，non-exact 也已有 `boundary-aware direct/shared-pack` 合同
- 但 `TargetLandingPlan + ResourceClosurePlan` 还没有扩成完整 fixed-point owner

结论：

- 结构不用推翻
- `boundary-aware direct` 和 `shared-pack` 已经升级成正式主线
- 剩余要补的是 shared-relay 的完整主线化，以及 general-shape 的真实性能闭环

#### 3. `WarpDecompositionPlan`

当前状态：

- 对当前 exact-tile direct path 已经可用
- 还明显绑定在 direct epilogue 上

为什么是 B 级：

- 它已经不只是几何壳子，已经带 `mmaGroupIds / accumulatorPackIds / epiloguePackIds`
- 但它还没有泛化到 relay/fallback/更复杂 landing

结论：

- 结构方向正确
- 后续需要从“只服务 direct pack”扩到“服务多 landing kind”

#### 4. `ResourceClosurePlan`

当前状态：

- 已经在做资源预算和 landing tradeoff
- 但当前 tradeoff 仍主要围绕 direct/shared-pack 两种简单分支

为什么是 B 级：

- 它已经进入 lowering 校验
- 但估算模型还不够宽，不足以支撑 split-k/persistent/general fallback

结论：

- 应继续扩，不应重写

#### 5. `SharedEncodingAttr` 的策略层

当前状态：

- shared encoding 的 attr/type 设计已经对了
- 非平凡 swizzle 参数已经能生成并被 lowering 真正消费
- 但 swizzle 策略本身还不够强

为什么是 B 级：

- shared layout 的结构已经是显式 owner
- 但 shared swizzle / bank-conflict 规避策略还没做强

结论：

- attr 结构是成熟的
- 策略层还没有成熟

### C级：目前还只是占位或半成品

这些部分不能算“当前框架已经具备的能力”，最多只能算后续扩展方向已经有痕迹。

#### 1. `split-k` 完整执行链

当前问题：

- `ProgramMappingPlan` 里有字段
- 但没有完整 reduction scratch owner
- lowering 也不接收

结论：

- 当前不能算具备

#### 2. `persistent` 完整执行链

当前问题：

- 计划层有字段
- 没有真正的 persistent work assignment / tile fetch loop owner
- lowering 不支持

结论：

- 当前不能算具备

#### 3. general-shape 的完整 late epilogue vector IO

当前问题：

- [2026-04-06 更新] 这条旧问题已经做完：
- late vector IO 已支持 `vector<2xf32>` / `vector<4xf32>`
- `boundaryAware=true` 已支持 `MaskedLoad/MaskedStore`
- 现在剩下的不是“late vector IO 不能表达”，而是 general-shape 的 runtime/perf 闭环还没完成

结论：

- 这部分代码能力已经不再是阻塞项
- 当前仍不能算 general-shape 已完成，因为 benchmark/硬件验收还没收口

## 三、Pass 成熟度分级

### A级：当前主线已经稳定

这些 pass 我认为已经可以作为当前 stage1 主链的稳定部分。

- `tb-verify-scope`
- `tb-attach-target-info`
- `tb-semanticize-matmul`
- `tb-build-layout-plan`
- `tb-build-transport-plan`
- `tb-rewrite-matmul-mainloop`
- `tb-cleanup-layout-conversions`
- `tb-build-mainloop-graph`
- `tb-regularize-k-loop`
- `tb-assign-latencies`
- `tb-schedule-loops`
- `tb-derive-waits`
- `tb-expand-pipeline`
- `tb-cleanup-pipeline`

成熟原因：

- 这些 pass 已经形成当前最重要的显式主线闭环
- 它们不是单纯挂 attr，而是真把主线推进到可 lower 的 `tb.pipeline_mainline`
- 这部分正是当前仓库和“只写后端模板”的最大区别

### A-级：对当前 strict exact-tile 主线已经稳定

- `tb-lower-pipeline-to-nvgpu`
- `tb-lower-epilogue-vector-io`

成熟原因：

- 对当前 exact-tile direct path，这两步已经跑通并且性能很好
- 它们已经不是“概念实现”，而是真正参与 benchmark 的 lowering 主线
- [2026-04-06 更新] `tb-lower-epilogue-vector-io` 已不再只服务 strict exact-tile，也已覆盖 boundary-aware / non-exact 合同

为什么不是完全 A 级：

- 它们的适用范围还是窄
- 一旦扩大到 boundary/shared-relay/split-k/persistent，就还不够

### B级：方向正确，但只覆盖当前一部分主线

- `tb-build-program-mapping-plan`
- `tb-build-c-register-plan`

原因：

- `tb-build-program-mapping-plan` 对 `tile/grouped_tile` 已经成熟，但 `split-k/persistent` 还没落到执行
- `tb-build-c-register-plan` 对 direct C path 很成熟，但 `warp/resource closure` 还明显绑定在 direct 主线

### C级：当前还不能视为完整能力

严格来说，当前 pass 链里没有单独的 `split-k lowering`、`persistent lowering`、`general-shape relay epilogue mainline` 这类 pass。

也就是说：

- 当前不是“这些 pass 质量不高”
- 而是“这些 pass 还没有被正式实现成主线能力”

---

## 四、当前最该保住的部分

如果后面继续做优化，最不该再推倒的部分是：

- `EncodingPlan / TransportPlan / AccumulatorPlan`
- `BufferModel -> LoopPlan -> LatencyPlan -> PipelinePlan -> AsyncPlan -> PipelineExpansion`
- `tb.pipeline_mainline`
- exact-tile 的 `tb-lower-pipeline-to-nvgpu` 主线

原因：

- 这些已经形成当前项目里最完整、最值钱的 owner-truth 主链
- 也是当前真实性能好的最核心原因之一

---

## 五、当前最需要继续扩的部分

如果后面继续做完整 matmul 框架，优先级建议如下：

1. general-shape 的 runtime/perf/benchmark 闭环
2. `shared-relay` 升级成正式主线
3. `EncodingPlan` 的 shared swizzle 策略做强
4. `ProgramMappingPlan` 的 `split-k`
5. `ProgramMappingPlan` 的 `persistent`
6. `WarpDecompositionPlan / ResourceClosurePlan` 从 direct 绑定扩展到多 landing kind

---

## 六、一句话判断

当前项目不是“数据结构还没建好，先别谈性能”，而是：

- 当前主线数据结构已经足够支撑强的 stage1 exact-tile matmul
- 当前主线 pass 也已经足够形成真实性能闭环
- 还没成熟的部分，主要是覆盖范围和分支完整度，不是主线骨架本身

也就是说：

- `主线已经成熟`
- `分支还要继续扩`
