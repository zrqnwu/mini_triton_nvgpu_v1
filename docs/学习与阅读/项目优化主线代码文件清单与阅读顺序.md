# 项目优化主线代码文件清单与阅读顺序

## 1. 这份文档要解决什么问题

这份文档回答三个非常实际的问题：

1. 你的项目大体上是怎样一步步优化 `tb.matmul` 的
2. 真正实现优化逻辑的代码文件是哪几个
3. 哪些文件主要是 pass 串联，哪些文件是辅助数据结构，应该按什么顺序读

本文基于当前仓库真实代码：

- `/home/zhangruiqi/mini_triton_nvgpu_v1`

---

## 2. 先给结论

如果只看整体结构，你这个项目的优化主线是：

```text
tb.matmul
  -> scope / target
  -> matmul 语义
  -> CTA/program 映射
  -> layout / encoding
  -> transport
  -> accumulator / epilogue / resource closure
  -> matmul mainloop rewrite
  -> buffer model
  -> loop regularization
  -> latency / schedule / waits
  -> pipeline expansion / cleanup
  -> pipeline_mainline
  -> NVGPU lowering
  -> NVVM / PTX
```

这里最重要的区分是：

1. `lib/Analysis/*.cpp` 里大多是真正“做优化决策”的地方
2. `lib/Transforms/*.cpp` 里很多文件主要是把这些决策挂成 pass、写回 attr、按顺序串起来
3. `LowerPipelineToNVGPU.cpp` 和 `LowerEpilogueVectorIO.cpp` 是最后把优化合同真正落到目标后端的地方

所以如果你问“优化代码主要在哪里”，答案优先是：

- `lib/Analysis/*.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`
- `lib/Transforms/LowerEpilogueVectorIO.cpp`

## 4. 第一类：真正实现优化逻辑的核心文件

这一组最重要。

这些文件真正负责：

- 生成语义合同
- 生成 layout/transport 合同
- 生成 register/epilogue 合同
- 建 buffer/resource graph
- 做 latency/schedule/wait/pipeline 推导
- 做最终 target lowering

下面按“执行顺序 + 重要程度”排序。

### 4.1 入口硬约束

#### 1. `lib/Analysis/KernelConfig.cpp`

作用：

- 解析 `tb.matmul` 的问题规模、tile、`numWarps`、`requestedStages`
- 给后续所有 pass 提供 kernel 入口配置

为什么重要：

- 没有它，后面所有分析都没有统一输入

#### 2. `lib/Analysis/TargetInfo.cpp`

作用：

- 生成目标 GPU 的能力信息
- 包括 `mma.sync`、`ldmatrix`、`cp.async`、shared/register 预算等

为什么重要：

- 这是所有 target-specific 优化是否合法的前提

### 4.2 语义与 CTA 映射

#### 3. `lib/Analysis/MatmulSemantics.cpp`

作用：

- 把 `tb.matmul` 变成显式 matmul 语义合同
- 固定 problem shape、tile shape、exact-tile、boundary 形态

为什么重要：

- 后面不是直接对高层 op 猜语义，而是消费这里给出的语义真相

#### 4. `lib/Analysis/ProgramMappingPlan.cpp`

作用：

- 决定 program/CTA 到 tile 的映射
- 决定 grouped launch / split-k / launch order 的主合同

为什么重要：

- CTA 外部分工必须先明确，后面的 CTA 内 layout/pipeline 才不会混乱

#### 5. `lib/Analysis/ReductionPlan.cpp`

作用：

- 给 split-k / reduction 相关行为提供合同

为什么重要：

- 即便当前 stage1 主线主要是 exact-tile，这个文件也是 program mapping 侧的重要辅助合同

#### 6. `lib/Analysis/PersistentWorkPlan.cpp`

作用：

- 给 persistent 相关 owner truth 提供数据结构

为什么重要：

- 即便当前 lowering 还有限制，数据结构层已经为后续扩展预留了 owner

### 4.3 layout 与搬运

#### 7. `lib/Analysis/EncodingPlan.cpp`

作用：

- 生成 blocked/shared/dot/accumulator encoding
- 固定 fragment 形状、lane ownership、`ldmatrix` 相关访存真相

为什么重要：

- 这是第一层真正接性能的文件
- 没有这层，就没有稳定的 tensor-core operand 布局

#### 8. `lib/Analysis/TransportPlan.cpp`

作用：

- 决定 A/B 从 global 到 shared 的 transport 方式
- 固定 vector bytes、transaction bytes、async eligibility、cache policy

为什么重要：

- `cp.async` / vectorized copy / cache 行为都建立在这里

### 4.4 C 路径与资源闭包

#### 9. `lib/Analysis/AccumulatorPlan.cpp`

作用：

- 定义 C accumulator 在寄存器里的 pack 组织和 lane access

为什么重要：

- 这是 C 在寄存器里“长什么样”的真相

#### 10. `lib/Analysis/EpiloguePlan.cpp`

作用：

- 定义 accumulator init/store 路径
- 区分 direct global vector 和 shared relay
- 定义 target landing

为什么重要：

- C 路径正确与否、是否会退化成 relay，都和这里强相关

#### 11. `lib/Analysis/EpilogueReorderPlan.cpp`

作用：

- 定义 epilogue reorder 的 owner truth

#### 12. `lib/Analysis/WarpDecompositionPlan.cpp`

作用：

- 定义 warp 如何覆盖 CTA tile
- 定义 warp 级 tile ownership

#### 13. `lib/Analysis/SharedWorkspacePlan.cpp`

作用：

- 定义 shared workspace 的分段、复用和 alias 策略

#### 14. `lib/Analysis/ResourceClosurePlan.cpp`

作用：

- 汇总 shared/register/workspace 等资源闭包
- 判断主线资源是否闭合

为什么这一组重要：

- 这几份文件共同决定了 C 路径、warp 分工、shared 工作区、资源预算是否自洽

### 4.5 matmul 专项主循环合同

#### 15. `lib/Analysis/MatmulRewritePlan.cpp`

作用：

- 把高层 matmul 改写成 tensor-core mainloop contract

为什么重要：

- 从这里开始，后续优化面对的不再是抽象 `tb.matmul`，而是一个已经明确 fragment/operand path 的 matmul 专项执行体

### 4.6 资源图、循环、时序、异步

#### 16. `lib/Analysis/BufferModel.cpp`

作用：

- 建立 backings / views / values / ops 的显式资源图

为什么重要：

- 这是后面 schedule/wait/pipeline 的核心输入
- 也是项目里最值得反复读的一份分析文件之一

#### 17. `lib/Analysis/LoopPlan.cpp`

作用：

- 把 main K-loop 规整成统一 owner

为什么重要：

- 后面的 pipeline pass 不必再从原始 IR 猜 loop 结构

#### 18. `lib/Analysis/LatencyPlan.cpp`

作用：

- 给每个 pipeline op 分配 latency、buffer distance、pipelineable 标记

为什么重要：

- 这是 schedule 的直接前提

#### 19. `lib/Analysis/PipelinePlan.cpp`

作用：

- 决定每个 op 的 stage、cluster、order

为什么重要：

- 这里开始真正进入时间维度优化

#### 20. `lib/Analysis/AsyncPlan.cpp`

作用：

- 推导 async producers、groups、waits、reuse fences

为什么重要：

- `cp.async` 对应的 group/wait 真相主要在这里

#### 21. `lib/Analysis/PipelineExpansion.cpp`

作用：

- 把抽象 schedule 展开成显式 pipeline cluster 序列

为什么重要：

- cleanup 和 lowering 消费的是这层显式化的 cluster 结果

#### 22. `lib/Analysis/PipelineReady.cpp`

作用：

- 给 cleanup/lowering 提供 compact ready summary

#### 23. `lib/Analysis/KernelContract.cpp`

作用：

- 把前面所有 plan 收口成 lowering 消费的总合同

为什么重要：

- target lowering 不是零散去 parse 各种 attr，而是通过这里统一读取合同

### 4.7 最终落地到 GPU IR

#### 24. `lib/Transforms/LowerPipelineToNVGPU.cpp`

作用：

- 读取 `tb.pipeline_mainline` 和 `KernelContract`
- 生成 `gpu` / `nvgpu` kernel IR

为什么重要：

- 这是优化真正落地成可执行 GPU IR 的核心文件
- 它不是前期“推导优化策略”的起点，但它是“让优化变成真实代码”的终点

#### 25. `lib/Transforms/LowerEpilogueVectorIO.cpp`

作用：

- 保住 direct-global C epilogue 的 vector load/store
- 防止最后阶段被 silently scalarize

为什么重要：

- 它负责守住最终 C epilogue 的 vector IO 质量

## 9. 最后用一句话总结

如果从“真正实现优化”的角度看，你的项目主要是：

1. 用 `lib/Analysis/*.cpp` 生成一层层显式优化合同
2. 用 `lib/Transforms/*.cpp` 按顺序把这些合同串起来
3. 用 `CleanupPipeline.cpp` 把合同封成 `tb.pipeline_mainline`
4. 用 `LowerPipelineToNVGPU.cpp` 和 `LowerEpilogueVectorIO.cpp` 把合同落到 GPU 后端

所以读项目时最重要的判断标准是：

- 想看“优化思想和推导”，优先读 `lib/Analysis`
- 想看“pass 主线和边界”，读 `lib/Transforms` 前半段
- 想看“优化最终怎么落成 GPU IR”，读 lowering 相关文件
