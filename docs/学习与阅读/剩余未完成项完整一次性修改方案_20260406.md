# 剩余未完成项完整一次性修改方案_20260406

本文档只回答两个问题：

1. 现在还没做好的地方，到底还剩哪些。
2. 如果不做局部补丁，而是一次性把剩余缺口按 Triton 思想收口，应该怎么改。

---

## 一、先把已经做完的地方钉死

这部分已经不是“待做项”，后续方案不能再回头把它们当主问题。

### 1. `shared swizzle` 不再是空壳

- `EncodingPlan` 已经不再固定输出 `perPhase=1 / maxPhase=1 / swizzlingByteWidth=0`
- `LowerPipelineToNVGPU` 已经把 shared physical layout 真正消费到 `cp.async` dst 和 `ldmatrix` src
- 所以“shared encoding 只有 attr、没有执行”这条已经关闭

### 2. `boundary-aware direct` 和 `shared-pack` 已经进入主线

- `EpiloguePlan` 现在已经能导出：
  - `RegisterPackGlobalVector`
  - `SharedPackThenGlobalVector`
- `CleanupPipeline` 已经接受 exact direct / boundary-aware direct / shared-pack 这三类 C landing 合同
- `LowerPipelineToNVGPU` 也已经有 shared-pack init/store 分支

### 3. `late vector IO` 旧限制已经解除

- 不再只接受 `exact-tile + vector<4xf32> + boundaryAware=false`
- 现在已经支持：
  - `vector<2xf32>`
  - `vector<4xf32>`
  - `boundaryAware=true` 时的 `MaskedLoad/MaskedStore`

### 4. `grouped_tile` 不再只是计划层占位

- `ProgramMappingPlan` 的 `grouped_tile` 已经能通过 strict lowering
- 当前真正还没落地的是 `split-k / persistent`

## 二、当前真正还没做好的地方

到 2026-04-06 为止，剩余问题集中在六类。

### 1. `split-k` 完整执行链没有落地

现状：

- `ProgramMappingPlan` 已有 `split_k` 相关字段
- 但没有 partial result owner
- 没有 reduction scratch owner
- 没有 split-k 的 cleanup/lowering 主线
- strict lowering 仍明确拒绝 `split-k`

根因：

- 现在的 mapping 只表达“program 如何编号”，没有把“多个 program 写同一 tile 时，中间结果放哪里、谁归约、何时同步”冻结成 owner truth

结论：

- 这不是调参问题
- 必须补一条完整的 split-k owner 链

### 2. `persistent` 完整执行链没有落地

现状：

- 计划层有 `persistent` 字段
- 但没有真实的 work assignment owner
- 没有 tile fetch loop owner
- 没有 persistent 对资源闭环的影响建模
- lowering 仍明确拒绝 `persistent`

根因：

- 当前 CTA/program 关系还是“一 program 对一 tile 或一 grouped launch slot”
- persistent 需要把“program 生命周期内反复取 tile 并消费”的行为前移成显式合同

结论：

- 这也不是小修能补的
- 必须新增 persistent work owner

### 3. general-shape 还没有真实性能闭环

现状：

- 合同和 lowering 现在已经能表达 `boundary-aware direct` / `shared-pack`
- 但当前公平 benchmark 脚本仍然硬编码 `exact_tile = true`
- 所以现在只有 exact-tile 的真实硬件闭环
- general-shape 还缺：
  - 独立 benchmark 驱动
  - 与 Triton 的公平对照口径
  - PTX/SASS/NCU 的收口证据

根因：

- 代码路径已经存在，但验证层还停留在 smoke/regression
- 所以当前不能把“能 compile”误当成“已经性能收口”

### 4. `ResourceClosurePlan` 还不够宽，也不够硬

现状：

- 现在只在 direct/shared-pack 两档之间做估算
- 还没有真正统一：
  - register pressure
  - A/B static shared
  - C landing shared
  - split-k scratch
  - persistent 额外状态

根因：

- 当前 `ResourceClosurePlan` 还是轻量 tradeoff summary
- 不是 Triton 式“所有关键资源都参与决策”的固定点 owner

影响：

- 后续一旦扩到 split-k/persistent/general fallback
- 现在这份 plan 就会变成“结构在，但不足以决定主线”

### 5. `SharedEncoding` 策略层还不够强

现状：

- 现在已经有真正的 swizzle 参数
- 但推导仍是偏启发式
- 还没有把：
  - operand role
  - ldmatrix fragment shape
  - async vector width
  - bank conflict 风险
  - grouped/persistent/split-k 带来的 shared 组织变化
  统一进更强的策略层

根因：

- 现在解决的是“不能表达”
- 还没有完全解决“如何稳定表达最优 shared physical layout”

### 6. stage1 shape closure 还不完整

现状：

- `KernelConfig` 仍限制 `num_warps in {1,4,8}`
- `cubin-format=mlir` 扩展 shape 的 sink 路径问题还没有形成正式闭环
- 部分非 exact 路径目前只有 compile/test，没有真实硬件验收

根因：

- 当前主线是“exact-tile stage1 强闭环”
- 但还没有扩成“完整 stage1 matmul shape closure”

## 三、这次完整方案的原则

这次方案不是继续补单个 bug，而是一次把剩余六类问题都收进统一结构。

### 原则 1

凡是以后会影响 lowering、性能或合法性的真相，都不能只挂在 `ProgramMappingPlan` 的枚举里，必须有对应 owner 结构。

### 原则 2

不能再让 `ResourceClosurePlan` 只做轻量摘要，而要让它真正决定：

- 哪条 landing 主线合法
- 哪条 mapping 主线合法
- 需要多少 static shared / extra shared / scratch
- 是否允许进入 persistent / split-k

### 原则 3

general-shape 不能只做到“能 lower”，必须同时补：

- contract
- cleanup
- lowering
- benchmark
- NCU/SASS 验收

### 原则 4

这次只做扩能力，不回滚已经完成的 exact-tile fast path。

## 四、一次性完整修改方案

下面按模块给出完整方案。

## 4.1 新增 `ReductionPlan`，把 split-k 做成正式 owner

新增文件：

- `include/tb/Analysis/ReductionPlan.h`
- `lib/Analysis/ReductionPlan.cpp`

修改文件：

- `include/tb/Analysis/ProgramMappingPlan.h`
- `lib/Analysis/ProgramMappingPlan.cpp`
- `lib/Transforms/CleanupPipeline.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`
- `lib/Transforms/Stage1Pipelines.cpp`

要做的事情：

1. 新增 `ReductionPlan`
   - 明确表达：
     - reduction kind
     - partial tile owner
     - scratch memory space
     - scratch shape
     - producer/consumer program relation
     - final reduction executor

2. `ProgramMappingPlan` 不再只写 `splitK = N`
   - 而是显式指向：
     - `mapping_kind = split_k`
     - `reduction_plan_id` 或等价显式 payload

3. cleanup 阶段必须校验 split-k 主线真相
   - 没有 `ReductionPlan` 就不允许进入 split-k mainline

4. lowering 侧补 split-k 路径
   - partial accumulator 写 scratch
   - final program 或单独 reduction phase 完成归约
   - direct/shared-pack epilogue 只作用在 final reduced tile 上

5. 资源估算必须把 reduction scratch 纳入 closure

完成后验收：

- strict lowering 不再拒绝 `split-k`
- `tb.pipeline_mainline` 上能看到 split-k owner truth
- 有 split-k smoke/regression
- 有 split-k fair benchmark

## 4.2 新增 `PersistentWorkPlan`，把 persistent 做成正式 owner

新增文件：

- `include/tb/Analysis/PersistentWorkPlan.h`
- `lib/Analysis/PersistentWorkPlan.cpp`

修改文件：

- `include/tb/Analysis/ProgramMappingPlan.h`
- `lib/Analysis/ProgramMappingPlan.cpp`
- `lib/Analysis/LoopPlan.cpp`
- `lib/Transforms/CleanupPipeline.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`

要做的事情：

1. 新增 `PersistentWorkPlan`
   - 表达：
     - program 取 tile 的顺序
     - tile fetch loop 边界
     - 单个 persistent program 的 tile batch
     - 完成条件

2. `ProgramMappingPlan` 的 `persistent=true` 不能再是布尔占位
   - 必须连带一份完整 `PersistentWorkPlan`

3. loop regularization / pipeline cleanup 要接受 persistent work loop
   - 不能再假设“一次 program 对一个 tile”

4. lowering 侧增加 persistent tile fetch 主线
   - `program_id -> first tile`
   - `while (hasWork)` 式 tile fetch
   - 每轮复用同一 CTA 内资源

5. 资源闭环要显式纳入 persistent 额外 live state

完成后验收：

- strict lowering 不再拒绝 `persistent`
- persistent path 有独立 regression
- 与 non-persistent path 共享同一 epilogue / async / shared owner 体系

## 4.3 把 `ResourceClosurePlan` 扩成真正的 fixed-point 资源 owner

修改文件：

- `include/tb/Analysis/ResourceClosurePlan.h`
- `lib/Analysis/ResourceClosurePlan.cpp`
- `lib/Transforms/CleanupPipeline.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`

要做的事情：

1. 扩展输入来源
   - 让它显式消费：
     - `ProgramMappingPlan`
     - `WarpDecompositionPlan`
     - `EpiloguePlan`
     - `ReductionPlan`
     - `PersistentWorkPlan`
     - `EncodingPlan`
     - `TransportPlan`

2. 扩展输出字段
   - 不再只有：
     - `estimatedAccumulatorRegs`
     - `estimatedEpilogueRegs`
     - `estimatedABShared`
     - `estimatedEpilogueShared`
   - 还要补：
     - `estimatedReductionScratch`
     - `estimatedPersistentState`
     - `estimatedTotalStaticShared`
     - `estimatedDynamicShared`
     - `estimatedTotalRegs`
     - `selectedMainlineKind`
     - `selectedCLandingKind`
     - `selectedBufferingKind`

3. cleanup 以后只允许 lowering 消费这份 closure 真相
   - 不允许 lowering 再自己猜资源取舍

4. 形成 rejection rule
   - 资源不闭环就直接拒绝进入该主线
   - 不再让 lowering 到末端再 silently fallback

完成后验收：

- `ResourceClosurePlan` 足以决定 direct/shared-pack/shared-relay/split-k/persistent 的合法性
- 后续不需要再回头改资源结构本身

## 4.4 把 `WarpDecompositionPlan` 从 direct-only 扩成多 landing owner

修改文件：

- `include/tb/Analysis/WarpDecompositionPlan.h`
- `lib/Analysis/WarpDecompositionPlan.cpp`
- `lib/Analysis/EpiloguePlan.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`

要做的事情：

1. 让 warp decomposition 不再只服务 direct pack
   - 需要显式补：
     - landing fragment group
     - shared-pack tile group
     - reduction partial group
     - persistent tile batch group

2. `EpiloguePlan` 不能再单独从 accumulator packs 推全部 landing 关系
   - 要改成：
     - accumulator ownership
     - warp decomposition
     - target landing legality
     三者共同收口

3. 这样后续 direct/shared-pack/shared-relay/split-k/persistent 都能复用同一套 warp owner truth

完成后验收：

- `WarpDecompositionPlan` 不再明显绑定 direct epilogue
- future pass 无需再引入新的 warp-local 第二真相

## 4.5 把 `EpiloguePlan` 收成三条正式主线

修改文件：

- `include/tb/Analysis/EpiloguePlan.h`
- `lib/Analysis/EpiloguePlan.cpp`
- `lib/Transforms/CleanupPipeline.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`
- `lib/Transforms/LowerEpilogueVectorIO.cpp`

这次要明确冻结三条 C landing：

1. `RegisterPackGlobalVector`
   - exact-tile direct 主线
   - boundary-aware 且 lane-local 可掩码时也允许

2. `SharedPackThenGlobalVector`
   - general-shape 主线
   - direct pack 无法 lane-local 关闭时进入

3. `SharedRelayThenGlobalVector`
   - 为后续更复杂 layout convert / split-k final reduction / persistent 复杂回写准备
   - 不再只做概念占位

要做的事情：

1. 给第三条 landing 补完整数据结构
2. cleanup 校验三条主线各自的 owner truth
3. lowering 全部按 landing kind 分流
4. `LowerEpilogueVectorIO` 继续只负责 final global vector IO
   - 不能再兼做 landing owner

完成后验收：

- epilogue 的复杂性被明确留在 landing contract
- late vector IO 只做最后一跳，不再背更多职责

## 4.6 把 `SharedEncoding` 的策略层做强，但不改 owner 边界

修改文件：

- `lib/Analysis/EncodingPlan.cpp`
- `lib/Transforms/LowerPipelineToNVGPU.cpp`
- 必要时新增 `include/tb/Analysis/SharedLayoutStrategy.h`
- 必要时新增 `lib/Analysis/SharedLayoutStrategy.cpp`

要做的事情：

1. 保持现有 `SharedEncodingAttr` / shared physical layout owner 不变
2. 只增强策略层
   - 输入增加：
     - operand role
     - ldmatrix tile count / transpose
     - async vector bytes
     - grouped/split-k/persistent 对 shared 组织的要求
   - 输出更稳定的：
     - order
     - perPhase
     - maxPhase
     - swizzlingByteWidth
     - 必要 padding / span 约束

3. 新增 bank-conflict oriented 验收
   - 不只看“有没有 swizzle”
   - 要看“shared physical layout 是否合理”

完成后验收：

- swizzle 不只是“有”
- 而是能形成稳定策略，不需要以后再推翻结构

## 五、实施顺序

这次不能打散，必须按依赖顺序一次收口。

1. 先补 `ReductionPlan`
2. 再补 `PersistentWorkPlan`
3. 再扩 `ResourceClosurePlan`
4. 再扩 `WarpDecompositionPlan`
5. 再把 `EpiloguePlan` 收成三条正式 landing 主线
6. 再增强 `SharedEncoding` 策略层
7. 再补 general-shape benchmark / NCU / SASS 闭环
8. 最后放开 stage1 shape closure 与 sink 问题

原因：

- `split-k / persistent` 是最大的 owner 缺口
- `ResourceClosurePlan` 必须在它们之后扩，不然仍会缺输入
- epilogue 和 swizzle 策略应建立在新的 mapping/resource 真相上
- benchmark 与 shape closure 必须放最后做统一验收

## 七、一句话结论

当前项目已经不是“主线骨架没搭起来”，而是：

- exact-tile 主线已经强
- general-shape/shared-pack 已经进入正式合同和 lowering
- 剩余真正没做完的，是 `split-k / persistent / resource closure / stronger swizzle strategy / general-shape 真实性能闭环 / shape closure`

这六件要一起从 owner 层收口，才是最后一轮真正完整的一次性方案。
