# 当前不足补足详细修改方案

更新时间：2026-04-06

适用项目：

- `/home/zhangruiqi/mini_triton_nvgpu_v1`

本文档只讨论当前项目剩余不足怎么补，不讨论已经成熟、已经形成闭环的 exact-tile 主线重写。

---

## 一、先说结论

当前项目不是“主线都还没搭起来”，而是：

- `single-CTA + exact-tile + cp.async + ldmatrix + mma.sync + direct-global-vector`
  这条主线已经成立，而且性能很好
- 剩下真正不足的地方，主要是“覆盖范围”和“分支完整度”

所以后续修改的原则必须是：

1. 保住当前 exact-tile fast path，不要把它改散
2. 继续把缺失分支补成正式主线，而不是在 lowering 里叠更多临时特判
3. 先收口 stage1 的完整 matmul 主线，再做 stage2 扩展

## 二、当前剩余不足，总结成 8 条

### 不足 1：shared encoding 结构已经有了，但 shared swizzle 还没真正做强

当前状态：

- `SharedEncodingAttr` 已经存在
- `EncodingPlan` 也已经显式挂 shared encoding
- 但 shared 构造值仍非常保守：
  - `perPhase = 1`
  - `maxPhase = 1`
  - `swizzlingByteWidth = 0`

这说明：

- 共享内存布局已经被做成显式 owner
- 但真正的 bank-conflict 规避策略还没有进入主线

影响：

- 对当前 exact-tile 小范围 case 不一定马上拖慢
- 但扩到更广 shape、更复杂 tile、更强 shared 压力时，这会成为和 Triton 拉开差距的点

### 不足 2：general-shape / boundary-aware direct epilogue 还没真正打通

当前状态：

- `EpiloguePlan` 已经能表达 `DirectGlobalVector`
- 但 late epilogue vector IO 还要求：
  - exact-tile
  - `vector<4xf32>`
  - `boundaryAware = false`

这说明：

- 现在 direct-global-vector 这条路只在“边界很干净”的条件下是正式主线
- 一旦进入 general-shape，就没有真正稳定的落地链

影响：

- 你现在不能说“stage1 一般形状也已经收口”
- 这会直接卡住后续推广到更真实的 matmul 场景

### 不足 3：shared-pack / shared-relay epilogue 还不是正式主线

当前状态：

- `EpiloguePlan` 类型里已经预留了 `SharedRelay`
- `TargetLandingPlan` 和 lowering 里也已经有 shared-pack 的部分骨架
- 但 cleanup 现在仍然只接受 direct-global epilogue

这说明：

- 当前真正完整闭环的还是 direct path
- relay/shared-pack 路线还没有进入“正式、受严格校验、可长期维护”的主线

影响：

- 一旦 direct path 不再干净，就会缺正式 fallback
- 项目在 exact-tile 以外的覆盖能力不足

### 不足 4：program mapping 只打通了 `tile / grouped_tile`

当前状态：

- `ProgramMappingPlan` 里已经有：
  - `SplitK`
  - `PersistentTile`
  - grouped launch
- 但 lowering 现在只接：
  - `tile`
  - `grouped_tile`
- 明确拒绝：
  - `split-k`
  - `persistent`

这说明：

- mapping 数据结构方向是对的
- 但更完整的 CTA/program 调度策略还没有落成真实执行主线

影响：

- 当前性能主要代表 exact-tile fast path
- 不能代表更广调度策略空间已经完成

### 不足 5：`WarpDecompositionPlan / ResourceClosurePlan` 还明显绑定在 direct path

当前状态：

- 这两个结构已经不只是占位了
- 也已经被 lowering 真实消费
- 但推导过程仍明显依赖 direct epilogue

这说明：

- 现在它们更像“exact-tile direct path 的资源/warp 收口”
- 还不是广义 matmul landing 决策层

影响：

- 后续一旦接 shared-pack/shared-relay，当前资源闭环不够宽

### 不足 6：验证范围还窄

当前状态：

- 当前强验证范围主要是：
  - single-CTA
  - exact-tile
  - 6 个代表 shape
- general-shape、边界 case、fallback、mapping 扩展，还没有形成统一验收链

影响：

- 当前可以很有把握地说“主线很强”
- 但还不能说“这个框架已经覆盖 stage1 的完整 matmul 空间”

### 不足 7：target lowering 过于集中，部分合同校验还泄漏在大 lowering 文件里

当前状态：

- `LowerPipelineToNVGPU.cpp` 现在承担了太多职责：
  - target lowering
  - contract legality check
  - shared 物理坐标推导
  - C path landing 分流
- 一部分本该在更早阶段暴露的错误，仍然要等到 target lowering 才报出来

这说明：

- 当前剩余问题不只是“再补几个 pass”
- 还包括“pass 边界和 lowering 边界还没有完全收口”

影响：

- 后续一旦补 general-shape、shared-pack、split-k/persistent，复杂度会继续往 target lowering 堆
- lowering 会越来越难维护
- 面试时也会更难解释“哪层负责什么”

### 不足 8：测试与 benchmark 基础设施还偏薄

当前状态：

- 当前已经有公平 benchmark 脚本，这是很好的起点
- 但仓库里还缺完整的：
  - lit / FileCheck 回归测试
  - 负例测试
  - pass 级 before/after 用例
  - 更系统的 general-shape / fallback / mapping case 样本

这说明：

- 你已经开始做性能工程
- 但“可持续重构”的测试安全网还没有补齐

影响：

- 以后做结构性重构时，风险会偏高
- 很多合同破坏还要靠人工读 IR 或手动 smoke 才能发现
- 会拖慢后续 general-shape、shared-pack、stage2 mapping 的推进

## 补充说明：这些不足不只属于 pass

很多人容易把当前项目剩余工作理解成“继续补几个 pass 就完了”，这不准确。

当前真正还要补的，至少分布在下面 5 类工程工作里：

### 1. IR / attr / type / op 设计

这里负责：

- 把 shared layout、program mapping、epilogue landing、pipeline owner 做成显式真相
- 确保后续 pass 和 lowering 不再私自猜

当前主要不足：

- shared swizzle 参数虽然有结构，但还没形成强策略
- direct / shared-pack / relay 的 owner 边界还没有完全收口成稳定主线

### 2. pass pipeline 与合同边界

这里负责：

- 哪个 pass 决定哪类真相
- 哪个 pass 只做 cleanup
- 哪个 pass 只做 lowering

当前主要不足：

- 部分 contract check 还埋在 target lowering
- cleanup / verify / lower 的职责边界还能更清晰

### 3. target lowering / codegen

这里负责：

- 把 `tb.pipeline_mainline`、`tb.epilogue_plan` 等合同真正落到 GPU / NVGPU / NVVM 形态

当前主要不足：

- C path 分支还没有完全按 landing 合同显式 dispatch
- shared encoding 的物理访问还没有完全统一消费 encoding owner

### 4. benchmark / profiling / regression

这里负责：

- 公平测量
- 波动控制
- 用 `ncu / nsys / PTX / SASS` 解释性能

当前主要不足：

- benchmark 范围仍主要围绕 exact-tile 主线
- 还缺更系统的 general-shape / fallback / mapping 扩展测量链

### 5. testing / verification 基础设施

这里负责：

- lit / FileCheck
- 负例测试
- pass 级回归测试
- 结构性重构后的安全网

当前主要不足：

- 这部分还没有像主线 contract 一样形成正式工程基础

## 三、修改总原则

### 原则 1：先补主线合同，再改 lowering

不要在 `TBLowerPipelineToNVGPU` 里继续堆 if/else 特判，让 lowering 末端代替上层做决策。

正确做法是：

- 先把 `EpiloguePlan / TargetLandingPlan / ProgramMappingPlan / EncodingPlan`
  这些合同补完整
- 再让 lowering 按合同分流

### 原则 2：保留 exact-tile fast path

当前 fast path 已经有真实性能结果，不能为了做 general-shape 把它改成“大一统慢路径”。

正确做法是：

- exact-tile direct path 保持 current fast path
- 其它路径作为正式分支加入

### 原则 3：stage1 和 stage2 分开收口

这里建议这样分：

- `stage1 完整收口`
  - shared swizzle 做强
  - general-shape / boundary-aware direct
  - shared-pack / shared-relay 主线化
  - warp/resource closure 泛化
- `stage2 扩展`
  - split-k
  - persistent
  - 更大范围调度策略

原因：

- `split-k / persistent` 会引入新的 reduction/work assignment owner
- 这已经不是“修补当前 stage1 主线”的量级

## 四、详细修改方案

## 第一组：先把 C path 收口成三条正式主线

### 4.1 重构 `EpiloguePlan` 的 landing 决策

修改文件：

- `include/tb/Analysis/EpiloguePlan.h`
- `lib/Analysis/EpiloguePlan.cpp`

当前问题：

- `deriveEpiloguePlan(...)` 主要只在构造 direct path
- `deriveTargetLandingPlan(...)` 也明显偏 exact-tile direct

目标：

- 把 C landing 正式拆成三条：
  1. `ExactDirectGlobalVector`
  2. `BoundaryAwareDirectGlobalVector`
  3. `SharedPackThenGlobalVector`

建议改法：

1. 保持 `DirectGlobalVectorPlan` 结构不推翻
2. 保持 `TargetLandingPlan` 作为 lowering 唯一分流真相
3. 扩大 `TargetLandingPlan.kind` 的使用范围
4. 用 `boundaryAware + scalarTail + sharedPackSlots + requiredSyncKind`
   表达差异，不要散落成一堆外层布尔字段

新增规则：

- exact-tile 且 direct pack 完整合法：
  - `TargetLandingKind::RegisterPackGlobalVector`
- non-exact-tile 但 direct path 仍合法：
  - 仍是 `RegisterPackGlobalVector`
  - 但 `boundaryAware = true`
- direct pack 不再合法，或者 shared-pack 明显更合理：
  - `TargetLandingKind::SharedPackThenGlobalVector`

验收标准：

- `tb.epilogue_plan` 不再只会导出 exact-tile direct 一种 landing
- non-exact-tile 下仍能导出合法的 direct/shared-pack landing

### 4.2 把 `TBLowerEpilogueVectorIO` 从 exact-only 变成真正的 late target owner

修改文件：

- `lib/Transforms/LowerEpilogueVectorIO.cpp`

当前问题：

- 只接受 `vector<4xf32>`
- 只接受 `boundaryAware = false`

目标：

- exact-tile fast path 继续走 `ld/st.global.v4.b32`
- boundary-aware path 改成 masked / predicated vector IO
- `vector_width` 由 op attr 决定，而不是 pass 内写死

建议改法：

1. 保留 fast path：
  - `vector_width == 4`
  - `boundaryAware == false`
  - 继续发 inline PTX `ld.global.v4.b32 / st.global.v4.b32`
2. 新增 boundary path：
  - 若 `boundaryAware == true`
  - 发逐 lane mask
  - 使用 vector 级装配/拆包，而不是直接整体报错
3. 支持非 `v4` 宽度：
  - 先支持 `v2` 和 `v4`
  - 不建议第一轮就做很宽泛的任意宽度

验收标准：

- exact-tile 现有 fast path 不回退
- general-shape 不再因为 late vector IO 直接 fail

### 4.3 cleanup 不再只允许 direct-global epilogue

修改文件：

- `lib/Transforms/CleanupPipeline.cpp`

当前问题：

- cleanup 直接拒绝非 direct epilogue

目标：

- cleanup 的职责应该是校验“landing 合同是否完整、pipeline 是否能 lower”
- 不应该把 epilogue 路径硬编码为 single-path

建议改法：

1. 去掉“只接受 direct-global epilogue”的硬判断
2. 改成：
  - exact-tile fast path：仍然强校验 direct path
  - 其它路径：校验 `TargetLandingPlan` 已完整
3. 对 shared-pack path 增加强校验：
  - shared tile rows/cols
  - shared pack slots
  - sync kind
  - vector width 对齐

验收标准：

- cleanup 可以产出 direct/shared-pack 两类 `tb.pipeline_mainline`

### 4.4 把 `LowerPipelineToNVGPU` 的 C path 改成显式 dispatch

修改文件：

- `lib/Transforms/LowerPipelineToNVGPU.cpp`

当前问题：

- 虽然已经有 shared-pack 的部分骨架
- 但主线结构仍然默认 direct path 先行，分支组织不够清晰

目标：

- 让 C path lowering 变成真正的“按 landing 合同分流”

建议改法：

拆成 3 个显式 helper：

1. `emitExactDirectEpiloguePath(...)`
2. `emitBoundaryAwareDirectEpiloguePath(...)`
3. `emitSharedPackEpiloguePath(...)`

保留当前 exact direct 逻辑：

- 不做大改
- 只抽函数

shared-pack path：

- 复用当前已有的 `usesCSharedPack` 基础骨架
- 但让它完全由 `TargetLandingPlan` 驱动
- 不再把它写成 direct path 的附属分支

验收标准：

- exact direct path 代码更清晰
- shared-pack path 成为正式 lowering 分支
- `validateLoweringContract(...)` 不再把非 direct path 都当异常

## 第二组：把 shared encoding 从“有结构”推进到“策略有效”

### 4.5 shared swizzle 参数推导真正进入 `EncodingPlan`

修改文件：

- `lib/Analysis/EncodingPlan.cpp`
- `include/tb/IR/TBAttrDefs.td`
- 如有必要，`lib/IR/TBAttrs.cpp`

当前问题：

- shared attr 结构是对的
- 但实际参数是“保守默认值”

目标：

- 根据硬件/元素字节数/`ldmatrix` 消费方式，推导 shared swizzle

建议改法：

1. 推导规则输入：
  - `TargetInfo.sharedBankBytes`
  - A/B element bytes
  - shared tile 几何
  - `ldmatrixTileCount`
  - operand transpose 情况
2. 推导结果输出到：
  - `perPhase`
  - `maxPhase`
  - `swizzlingByteWidth`
3. 第一轮目标：
  - 先覆盖当前 `fp16` A/B shared tile 的主线情况
  - 不要求一开始就完全一般化

重要提醒：

- 只改 attr 构造是不够的
- lowering 的 shared 物理坐标 materialization 也必须消费这些参数

验收标准：

- shared encoding 不再固定是 `1/1/0`
- shared 访问逻辑与 encoding 合同一致

### 4.6 lower 侧 shared 物理访问统一消费 shared encoding

修改文件：

- `lib/Transforms/LowerPipelineToNVGPU.cpp`

目标：

- shared physical row/col 计算统一走 shared encoding helper
- 不允许 lowering 对 shared layout 再做一套私有假设

验收标准：

- shared layout 真正变成“plan owner”
- 后续 shared swizzle 调整不会要求再改多套 lowering 逻辑

## 第三组：泛化 warp/resource 收口

### 4.7 泛化 `WarpDecompositionPlan`

修改文件：

- `lib/Analysis/WarpDecompositionPlan.cpp`
- `include/tb/Analysis/WarpDecompositionPlan.h`

当前问题：

- 现在仍明显要求 direct epilogue

目标：

- 让它表达“warp 覆盖 + mma group + accumulator pack + epilogue pack”
- 而不是“只服务 direct epilogue”

建议改法：

1. 保留当前 `ownerScope / warps / pack ids`
2. 抽象 epilogue pack 来源：
  - direct pack
  - shared-pack slot / relay pack
3. 增加 landing kind 对应的 pack role 标识

验收标准：

- shared-pack path 下仍能导出合法 warp decomposition

### 4.8 泛化 `ResourceClosurePlan`

修改文件：

- `lib/Analysis/ResourceClosurePlan.cpp`
- `include/tb/Analysis/ResourceClosurePlan.h`

当前问题：

- 预算估计还主要围绕 direct/shared-pack 简单二分

目标：

- 让它成为真正的 landing / buffering 预算收口层

建议改法：

1. 保留当前字段，不推翻
2. 增加：
  - landing candidate summary
  - shared-pack / direct / relay 的预算比较
3. 当前至少先覆盖：
  - exact direct
  - boundary direct
  - shared-pack

验收标准：

- lowering 校验的是“资源闭环正确”
- 而不是“资源闭环只服务 direct”

## 第四组补充：工程收口与验证体系补齐

这一组非常重要，因为它补的是“项目不只是功能主线，还要有工程可维护性”。

### 4.9 新增 `tb-verify-pipeline-mainline-contract`

修改文件：

- `include/tb/Transforms/Passes.td`
- 新增 `lib/Transforms/VerifyPipelineMainlineContract.cpp`
- `lib/Transforms/Stage1Pipelines.cpp`

目标：

- 在 `tb-cleanup-pipeline` 之后、`tb-lower-pipeline-to-nvgpu` 之前
- 把本该提前暴露的 mainline contract 问题，在 lowering 前报清楚

建议检查项：

1. `tb.pipeline_mainline` cluster ordinal 连续
2. 每个 pipeline op 恰好被 cluster 覆盖一次
3. async producer / wait group 一一对应
4. wait anchor 合法
5. `pipeline_ready` 摘要与显式 cluster 真相一致
6. exact-tile async 主线没有退化成非法 producer

为什么这一步值得单独做：

- 这不是“为了多写一个 pass”
- 而是为了把一部分 target lowering 的隐含前提前移成显式 contract verifier

验收标准：

- 同类错误不再主要等到 `TBLowerPipelineToNVGPU` 才暴露
- mainline contract 的错误信息更早、更清楚

### 4.10 拆分 `LowerPipelineToNVGPU` 的职责

修改文件：

- `lib/Transforms/LowerPipelineToNVGPU.cpp`

目标：

- 降低单文件过大、职责混杂的问题
- 让 verifier / legality / lowering dispatch / materialization 分层更清楚

建议改法：

1. 抽离 contract validation helper
2. 抽离 C path epilogue dispatch helper
3. 抽离 shared physical coordinate helper
4. 保持 public pipeline 行为不变，不做语义层回退

验收标准：

- lowering 主要做“消费合同并翻译”
- 不再继续堆更多“本该提前决定”的 if/else

### 4.11 建立正式 lit / FileCheck / 负例测试骨架

修改文件：

- 新增 `test/` 目录
- 新增 lit 配置
- 新增 pass 级和 lowering 前 contract 级测试样例

目标：

- 把当前“主要靠 smoke + benchmark + 人工读 IR”的状态，升级成正式回归链

第一轮最少补这几类测试：

1. `tb-cleanup-layout-conversions` 正例/反例
2. `tb-cleanup-pipeline` 正例/反例
3. `tb-verify-pipeline-mainline-contract` 正例/负例
4. `tb-lower-epilogue-vector-io` exact-tile / boundary-aware 两组测试
5. `ProgramMappingPlan` 的 tile/grouped_tile 负例测试

验收标准：

- 关键合同重构不再只靠人工 smoke
- 后续扩 general-shape / shared-pack 时有最小安全网

### 4.12 扩 benchmark 与 profiling 用例矩阵

修改文件：

- `tools/bench_stage1_fair.py`
- 新增或补充 shape 配置文件
- 如有需要，新建 `tools/profile_stage1_ncu.py` 或 shell 脚本

目标：

- benchmark 不再只代表“当前 exact-tile 最强主线”
- 逐步覆盖 exact-tile、non-exact-tile、boundary-aware、grouped mapping

第一轮建议补：

1. exact-tile 对照 shape
2. non-exact-tile 边界 shape
3. grouped launch shape
4. direct path 与 shared-pack path 对照 shape

验收标准：

- 修改方案不只是“能编译”
- 还能更系统地解释：为什么快、为什么慢、哪里是合同问题、哪里是资源问题

## 五、实施顺序

必须按这个顺序做：

1. `EpiloguePlan / TargetLandingPlan` 泛化
2. `LowerEpilogueVectorIO` 泛化
3. `CleanupPipeline` 接受多 landing kind
4. `LowerPipelineToNVGPU` 的 C path dispatch 重构
5. `EncodingPlan` shared swizzle 策略
6. `LowerPipelineToNVGPU` shared encoding 消费统一化
7. `WarpDecompositionPlan / ResourceClosurePlan` 泛化
8. 新增 `tb-verify-pipeline-mainline-contract`
9. 建立 lit / FileCheck / 负例测试骨架
10. 扩 benchmark / profiling 用例矩阵
11. 在上面全部收口后，再单开一轮做 `split-k / persistent`

不要打乱顺序，原因很明确：

- 如果先做 `split-k/persistent`，你会把当前还没补完的 stage1 主线问题带进更复杂 mapping
- 后面定位问题会非常痛苦
- 如果不先补 verifier / lit / benchmark 基础设施，后面每一轮重构都会过度依赖人工兜底

---

## 六、验收标准

### 第一批：stage1 收口验收

要同时满足：

1. exact-tile 当前 6 个 shape 性能不明显回退
2. non-exact-tile smoke case 能 compile + run
3. `tb.epilogue_plan` 不再只会导出 exact direct path
4. `tb.pipeline_mainline` 能承载 direct/shared-pack 两类 landing
5. shared encoding 不再固定是 `1/1/0`
6. late epilogue vector IO 不再直接拒绝 `boundaryAware = true`
7. mainline contract 破坏能在 lowering 前被 verifier 抓住
8. 关键 pass 至少有 lit / FileCheck 正例与负例

### 第二批：stage2 扩展验收

要同时满足：

1. `split-k` 有完整 reduction scratch owner
2. `persistent` 有完整 work assignment owner
3. lowering 不再拒绝 `split-k/persistent`
4. 新 mapping path 有独立 benchmark 和 correctness 验收

---

## 七、哪些地方不要动

以下内容不建议再推倒：

- `BufferModel -> LoopPlan -> LatencyPlan -> PipelinePlan -> AsyncPlan -> PipelineExpansion`
- `tb.pipeline_mainline`
- 当前 exact-tile direct fast path 的主要 lowering 结构
- `KernelContract` 的“集中解析 lowering 合同”思路

理由：

- 这些已经是当前项目最成熟、最接近 Triton 原理、且已经有性能闭环的一部分

---

## 八、一句话总结

当前项目真正该补的，不是“把所有优化技巧再堆一遍”，而是把下面五件事收口：

1. `C path` 从单一路线升级成正式多分支主线
2. `shared swizzle` 从结构存在升级成端到端有效
3. `general-shape / boundary-aware` 进入 stage1 正式覆盖
4. verifier / lit / benchmark 形成正式工程基础
5. `split-k / persistent` 作为 stage2 扩展独立落地

做完这几件事，你的项目就会从“很强的 exact-tile 原型”升级成“结构完整、可扩展、工程边界清楚、真正适合面试讲完整故事的 matmul 编译器项目”。
