## 2. 先给一个总判断

要先分清楚三类文件。

### 2.1 真正实现优化逻辑的文件

主要在：

- `lib/Analysis/*.cpp`

这些文件里真正决定：

- 语义怎么解释
- program/CTA 怎么映射
- layout/encoding 怎么建
- transport 怎么选
- accumulator/epilogue 怎么组织
- buffer graph 怎么建
- latency/schedule/wait 怎么推
- pipeline 怎么展开

也就是说，这里才是“优化决策真正发生的地方”。

### 2.2 pass 串联和合同封口文件

主要在：

- `lib/Transforms/*.cpp`

这些文件很多做的是：

- 调 `derive*Plan(...)`
- 把分析结果挂成 attr
- 校验 pass 输入是否齐全
- 把多个合同收口成更低层 owner
- 最后做 lowering

这里很重要，但它更多是“编排和封口”，不是最核心的分析决策本体。

### 2.3 数据结构和合同定义文件

主要在：

- `include/tb/Analysis/*.h`

这些文件不直接做优化算法，但它们决定：

- 每一层优化在表达什么
- 上一层给下一层交什么真相
- 为什么项目可以一步步优化而不是乱改

## 4. 第一组：前置边界与高层语义

这一组先建立“这个 kernel 究竟是什么、当前硬件支不支持”。

### 4.1 必读文件

1. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/KernelConfig.cpp`
2. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/TargetInfo.cpp`
3. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/MatmulSemantics.cpp`
4. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/VerifyScope.cpp`
5. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/AttachTargetInfo.cpp`
6. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/SemanticizeMatmul.cpp`

### 4.2 这一组在做什么

- `KernelConfig.cpp`
  读取 `tb.matmul` 和 module attrs，形成 kernel 问题规模、tile、num-warps、requested-stages 等入口配置。

- `TargetInfo.cpp`
  根据 `gpu-arch` 生成硬件能力合同，比如是否支持 `cp.async`、`ldmatrix`、`mma.sync`、threads per warp、共享内存预算等。

- `MatmulSemantics.cpp`
  把高层 `tb.matmul` 变成明确的 matmul 语义：problem shape、tile shape、exact-tile、boundary 情况。

- `VerifyScope.cpp`
  检查当前 `tb.matmul` 是否落在 stage1 支持范围内。

- `AttachTargetInfo.cpp`
  把 target info 和关键 module attr 固定到 module 上，确保后面不再靠默认值猜。

- `SemanticizeMatmul.cpp`
  调 `deriveMatmulSemantics(...)`，把语义结果挂到 op 上。

### 4.3 重要程度

这一组的重要性非常高，但它更偏“前提固定”，还不是直接提速。

如果没有这组：

- 后面的 layout、transport、schedule 没有可靠输入
- 很多优化只能靠隐式假设

## 5. 第二组：program 映射与物理布局

这一组开始进入真正的优化基础层。

### 5.1 必读文件

1. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/ProgramMappingPlan.cpp`
2. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/ReductionPlan.cpp`
3. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/PersistentWorkPlan.cpp`
4. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/EncodingPlan.cpp`
5. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/TransportPlan.cpp`
6. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/BuildProgramMappingPlan.cpp`
7. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/BuildLayoutPlan.cpp`
8. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/BuildTransportPlan.cpp`

### 5.2 这一组在做什么

- `ProgramMappingPlan.cpp`
  决定一个 program/CTA 负责哪个 tile，launch order 怎样，grouped launch 怎么表达。

- `ReductionPlan.cpp`
  表达 split-k 或归约相关的跨 program 合同。

- `PersistentWorkPlan.cpp`
  表达 persistent tile-fetch / resident program 语义。

- `EncodingPlan.cpp`
  这是第一层真正的物理合同。它决定 blocked/shared/dot/accumulator encoding，fragment 形状，以及 lane 到 operand fragment 的关系。

- `TransportPlan.cpp`
  决定 A/B 从 global 到 shared 怎样搬运，vector width 多大，能不能 async，cache policy 怎样。

### 5.3 重要程度

这一组是 stage1 里最关键的基础优化层之一。

原因是：

- 没有 `ProgramMappingPlan`，你不知道 CTA 负责哪块 tile
- 没有 `EncodingPlan`，你不知道 `ldmatrix` / `mma.sync` 的 lane 对齐真相
- 没有 `TransportPlan`，你不知道 async copy 是否成立

也就是说，很多真正影响性能的后续 pass，都建立在这组结果之上。

## 6. 第三组：C 路径、warp 拆分和资源闭包

这一组在决定 C accumulator、epilogue、warp 内分工和资源预算。

### 6.1 必读文件

1. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/AccumulatorPlan.cpp`
2. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/EpiloguePlan.cpp`
3. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/EpilogueReorderPlan.cpp`
4. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/WarpDecompositionPlan.cpp`
5. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/SharedWorkspacePlan.cpp`
6. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/ResourceClosurePlan.cpp`
7. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/BuildCRegisterPlan.cpp`

### 6.2 这一组在做什么

- `AccumulatorPlan.cpp`
  决定 C accumulator 在寄存器里的 pack 形态、lane access、寄存器占用。

- `EpiloguePlan.cpp`
  决定 C init/store 路径是 direct global vector 还是 shared relay，以及 target landing 真相。

- `EpilogueReorderPlan.cpp`
  如果 direct path 不够，需要表达 CTA shared reorder 的合同。

- `WarpDecompositionPlan.cpp`
  决定 warp 如何覆盖 CTA tile，以及 warp 拥有哪些 mma group / accumulator pack / epilogue pack。

- `SharedWorkspacePlan.cpp`
  统一表达 shared workspace 各段怎么分配、哪些环节复用。

- `ResourceClosurePlan.cpp`
  把寄存器、shared、workspace、buffering 等资源闭包成一个统一预算判断。

### 6.3 重要程度

这一组很重要，尤其是你后面做性能排查时：

- 如果 C path 走错
- 如果 workspace/relay 结构不合理
- 如果寄存器/shared 预算错了

问题通常就在这一组里。

这一组决定“主循环到底长什么样”。

### 7.1 必读文件

1. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/MatmulRewritePlan.cpp`
2. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/BufferModel.cpp`
3. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/LoopPlan.cpp`
4. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/RewriteMatmulMainloop.cpp`

6. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/RegularizeKLoop.cpp`

### 7.2 这一组在做什么

- `MatmulRewritePlan.cpp`
  把高层 matmul 变成 tensor-core mainloop contract。

- `BufferModel.cpp`
  建立完整资源图：
  - backings
  - views
  - values
  - ops

- `LoopPlan.cpp`
  把 main K-loop 规整成一个稳定 owner。

### 7.3 重要程度

这一组是整个项目“从语义化走向可调度化”的关键转折点。

尤其是 `BufferModel.cpp`，它非常值得重视，因为后面 latency、schedule、async wait 都是建立在它上面。

如果只选一个必须啃透的 analysis 文件，`BufferModel.cpp` 是候选之一。

## 8. 第五组：真正的时序优化

这一组是 stage1 里“最像流水线优化本体”的部分。

### 8.1 必读文件

1. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/LatencyPlan.cpp`
2. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/PipelinePlan.cpp`
3. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/AsyncPlan.cpp`
4. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/PipelineExpansion.cpp`
5. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/AssignLatencies.cpp`
6. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/ScheduleLoops.cpp`
7. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/DeriveWaits.cpp`
8. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/ExpandPipeline.cpp`

### 8.2 这一组在做什么

- `LatencyPlan.cpp`
  给每个 pipeline op 估计 latency、buffer distance、是否可 pipeline。

- `PipelinePlan.cpp`
  决定 op 放在哪个 stage、哪个 cluster、什么顺序。

- `AsyncPlan.cpp`
  推导 async producer、async group、wait 和 reuse fence。

- `PipelineExpansion.cpp`
  把抽象 schedule 展开成显式 cluster 流。

### 8.3 重要程度

这一组直接决定：

- copy 和 compute 能不能 overlap
- wait 是否放对
- async group 是否合理
- stage buffer 是否能安全复用

如果你以后看性能差距，这一组往往就是最核心的排查区。

## 9. 第六组：合同封口与 lowering

这一组不是前面那种 analysis 决策层，但它是“优化落地”的最后关口。

### 9.1 必读文件

1. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Analysis/KernelContract.cpp`
2. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/CleanupPipeline.cpp`
3. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/VerifyPipelineMainlineContract.cpp`
4. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/LowerPipelineToNVGPU.cpp`
5. `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/LowerEpilogueVectorIO.cpp`

### 9.2 这一组在做什么

- `KernelContract.cpp`
  把 lowering 所需的所有合同统一收口。

- `CleanupPipeline.cpp`
  校验前面生成的 async mainline 是否完整，然后把 `tb.matmul` 替换成显式的 `tb.pipeline_mainline`。

- `VerifyPipelineMainlineContract.cpp`
  在 lowering 前做最后一轮一致性校验。

- `LowerPipelineToNVGPU.cpp`
  真正读取 `KernelContract` 和显式 pipeline cluster，发射 GPU/NVGPU IR。

- `LowerEpilogueVectorIO.cpp`
  保住 direct-global epilogue 的 vector IO，不让后端把它静默标量化。

### 9.3 重要程度

如果你的目标是理解“优化怎么真正落到 GPU IR”，这组必须看。

如果你的目标是理解“优化决策本身是怎么来的”，那它排在 analysis 层之后。
