# 补足过程先学内容与顺序

更新时间：2026-04-06

适用项目：

- `/home/zhangruiqi/mini_triton_nvgpu_v1`

本文档回答的问题不是“所有相关知识都学一遍”，而是：

- 为了把当前项目的剩余不足补起来，哪些内容最值得先学
- 学到什么程度就够用
- 每一块学习完以后，马上能改项目里的哪一部分

---

## 一、学习原则

### 原则 1：只学和下一步代码修改直接相关的内容

不要先把 Triton、MLIR、CUDA、GPU 架构全学完再动手。那样周期太长，而且你很容易学散。

正确方式是：

- 先确定下一轮要改哪一块
- 再只补这一块直接需要的知识
- 学完立刻回项目里落代码

### 原则 2：优先学“为什么要这么设计”，再学“具体 API 怎么写”

当前项目最大的价值不是会不会调一个 pass 注册函数，而是你能不能解释：

- 为什么 owner 要放在这里
- 为什么这层合同不能留给 lowering 猜
- 为什么 exact-tile 很强但 general-shape 还没收口

### 原则 3：学习顺序要跟改造顺序一致

建议顺序：

1. 先学 stage1 收口必需知识
2. 再学 stage2 扩展知识

不要一开始就冲 `split-k / persistent`，那会把精力从当前最该收口的主线带偏。

## 二、第一阶段：马上就该先学的内容

这些是你下一轮补足 current stage1 缺口时，最值得先学的内容。

## 1. MLIR 里 attr / type / op / pass 的职责边界

为什么先学这个：

- 你当前项目的核心价值就是“把真相放进正确的 owner”
- 后续补 `shared swizzle`、`general-shape epilogue`、`shared-pack` 时，如果对 MLIR 职责边界不清楚，很容易又写回“lowering 末端猜一次”

要学到什么程度：

- 明白什么应该放 attr/type
- 明白什么应该放 analysis contract
- 明白什么应该进显式 IR op
- 明白 pass 应该做“决策”还是“翻译”

你学完后能直接改的地方：

- `EncodingPlan`
- `TargetLandingPlan`
- `tb.pipeline_mainline`
- `LowerPipelineToNVGPU`

学习目标：

- 能自己回答一句话：
  “这条真相为什么应该放在这一层，而不是下一层再猜？”

## 2. Triton matmul 的真实编译顺序

为什么要学：

- 你后面所有补足，都要尽量继续对齐 Triton 思想
- Triton 不是“先写 async，再补结构”，而是：
  - 先语义化
  - 再 layout/program mapping
  - 再 matmul rewrite
  - 再 pipeline
  - 再 target lowering

要学到什么程度：

- 理解 Triton 为什么先定 layout/CTA/fragment truth
- 理解为什么不能让 executor / lowering 再回头做上层决策

你学完后能直接改的地方：

- `ProgramMappingPlan`
- `EncodingPlan`
- `MatmulRewritePlan`
- `CleanupPipeline`

学习目标：

- 能清楚讲出：
  “当前项目哪些地方已经是 Triton 思想，哪些地方只是有壳子还没落地。”

## 3. Ampere 上的 `cp.async / ldmatrix / mma.sync` 真正约束

为什么要学：

- 当前 exact-tile 主线能快，靠的就是这条链
- 后续补 general-shape/shared swizzle 时，不能把这条快路径破坏掉

要学到什么程度：

- 明白 `cp.async` 的最小/偏好字节约束
- 明白 `ldmatrix` 为什么要求特定 shared 物理布局
- 明白 `mma.sync m16n8k16` 的 fragment 组织

你学完后能直接改的地方：

- `EncodingPlan`
- `TransportPlan`
- `LowerPipelineToNVGPU`

学习目标：

- 不只是会说名词，而是知道：
  - A/B shared tile 为什么要那样布
  - 为什么 `ldmatrix` 和 shared swizzle 必须统一看

## 4. shared memory swizzle / bank conflict

为什么要学：

- 这是你当前最明确的剩余不足之一
- 也是你最容易从“结构有了”走到“策略真的有效”的一块

要学到什么程度：

- 明白 shared bank conflict 是怎么来的
- 明白 `perPhase / maxPhase / swizzlingByteWidth` 是干什么的
- 明白 Triton shared encoding 为什么要把这些做成显式参数

你学完后能直接改的地方：

- `SharedEncodingAttr`
- `EncodingPlan.cpp`
- `LowerPipelineToNVGPU.cpp` 里 shared 物理访问

学习目标：

- 学完以后，你应该能自己写出一句判断：
  “这个 shared layout 为什么比 `1/1/0` 更合理。”

## 5. direct epilogue 与 relay/shared-pack epilogue 的区别

为什么要学：

- 当前项目里 C path 是下一轮最重要的收口点
- 如果不理解这两种 path 的根本区别，容易把 shared-pack 误做成 direct path 的补丁

要学到什么程度：

- 明白 direct-global-vector 什么时候最优
- 明白 shared-pack / relay 什么时候变成必须
- 明白它们在 owner 层应该怎么区分

你学完后能直接改的地方：

- `EpiloguePlan`
- `TargetLandingPlan`
- `CleanupPipeline`
- `LowerPipelineToNVGPU`

学习目标：

- 能清楚解释：
  “为什么 exact-tile current fast path 很快，但 general-shape 还需要 shared-pack/relay。”

## 6. 边界处理与 masked vector IO

为什么要学：

- 当前 late epilogue vector IO 还只接受 non-boundary exact-tile
- 你要补 general-shape，就必须处理边界

要学到什么程度：

- 明白 boundary-aware vector load/store 的几种做法：
  - predicated
  - masked vector
  - scalar tail
- 明白什么时候应该保 vector，什么时候不得不拆 tail

你学完后能直接改的地方：

- `LowerEpilogueVectorIO`
- `EpiloguePlan`
- `TargetLandingPlan`

学习目标：

- 学完后能明确回答：
  “为什么 general-shape 不能简单复用 exact-tile 的 late vector IO。”

## 7. 资源估计与 occupancy 基础

为什么要学：

- `WarpDecompositionPlan / ResourceClosurePlan` 后面要从 direct 绑定扩成更广 landing 决策层

要学到什么程度：

- 理解寄存器、static shared、dynamic shared 的基本预算关系
- 理解为什么某些 landing tradeoff 不是只看指令数，还要看资源

你学完后能直接改的地方：

- `ResourceClosurePlan`
- `LowerPipelineToNVGPU` 中的 validation / landing dispatch

学习目标：

- 学完以后你能更自然地判断：
  “这个 path 慢，是因为逻辑不对，还是资源 tradeoff 不对。”

## 8. lit / FileCheck / 负例测试基础

为什么这块要尽早学：

- 你后面不只是补功能，还会做结构性重构
- 如果没有 pass 级回归测试和负例测试，很难放心地重构 verifier、cleanup、lowering 边界

要学到什么程度：

- 明白 lit 测试基本组织方式
- 明白 FileCheck 怎么写 pass 前后断言
- 明白负例测试应该验证什么样的错误信息

你学完后能直接改的地方：

- 新建 `test/` 目录
- `tb-cleanup-layout-conversions` 测试
- `tb-cleanup-pipeline` 测试
- `tb-verify-pipeline-mainline-contract` 测试

学习目标：

- 学完以后你要能自己搭出最小测试骨架，而不是继续只靠 examples 和手动 smoke

## 9. benchmark / profiling / regression 口径

为什么这块要尽早学：

- 你已经有 benchmark 脚本，但后面要扩 general-shape、shared-pack、mapping 时，不能只报一个 runtime 数字
- 你要能解释性能变化到底是：
  - 合同变了
  - lowering 变了
  - 资源占用变了
  - 还是测量口径出问题

要学到什么程度：

- 明白 benchmark 如何做到公平
- 明白 `ncu / nsys` 最该先看哪些指标
- 明白怎样把 PTX / SASS / profiler 指标和 runtime 结果串起来

你学完后能直接改的地方：

- `bench_stage1_fair.py`
- 后续的 `profile_stage1_ncu.py` 或 shell 脚本
- 文档里的性能归因报告

学习目标：

- 学完以后，你要能把“它为什么快/慢”讲成一条完整证据链，而不是只给一个跑分

## 三、第二阶段：在第一阶段收口后再学的内容

这些内容很重要，但不建议抢在前面学。

## 10. `split-k` 的 reduction 设计

为什么放第二阶段：

- 它会引入新的 buffer role、partial accumulation、reduction scratch
- 这已经超过当前 stage1 主线收口的范围

要学到什么程度：

- 明白 split-k 的几种 reduction 形式：
  - serial
  - parallel
  - workspace based
- 明白 partial C 和 final reduction 的 owner 应该在哪里

你学完后能直接改的地方：

- `ProgramMappingPlan`
- `BufferModel`
- `LowerPipelineToNVGPU`

## 11. persistent kernel / work assignment

为什么放第二阶段：

- 这不是简单换个 launch order
- 需要重新思考 CTA 怎样取活、什么时候退出

要学到什么程度：

- 明白 persistent kernel 解决什么问题
- 明白 tile fetch loop 和普通 CTA tile 映射的差别

你学完后能直接改的地方：

- `ProgramMappingPlan`
- `LowerPipelineToNVGPU`

## 12. autotune / config search

为什么放第二阶段：

- 当前最先缺的不是 autotune，而是结构覆盖
- 如果主线分支还没齐，先上 autotune 只是把问题藏起来

要学到什么程度：

- 明白哪些参数适合调
- 明白 autotune 的前提是合法主线已经比较完整

你学完后能直接改的地方：

- benchmark harness
- config selection layer

## 五、每学完一块，项目里立刻该做什么

- 审一遍 `EncodingPlan / TransportPlan / MatmulRewritePlan / LowerPipelineToNVGPU`

立刻做：

- 改 `EncodingPlan` 的 shared swizzle 参数推导
- 改 lowering 的 shared 物理访问消费方式

### 学完 5-6 之后

立刻做：

- 改 `EpiloguePlan`
- 改 `TargetLandingPlan`
- 改 `LowerEpilogueVectorIO`
- 改 `CleanupPipeline`
- 改 `LowerPipelineToNVGPU` 的 C path dispatch

### 学完 7 之后

立刻做：

- 泛化 `ResourceClosurePlan`
- 让 landing 选择不再只是 direct 绑定

### 学完 8-9 之后

立刻做：

- 搭 `test/` 与 lit / FileCheck 骨架
- 给关键 pass 补正例和负例
- 扩 benchmark / profiling 用例矩阵
- 让后续重构不再主要靠人工 smoke

### 学完 10-11 之后

立刻做：

- 进入 stage2
- 扩 `split-k / persistent`

## 七、一句话总结

你现在不需要“从零开始再系统学一遍 GPU 编译器”，而是应该：

- 围绕当前项目剩余缺口，按顺序补知识
- 每补一块，就立刻把它落到当前代码里

最优学习策略不是“大而全”，而是：

- `学最接近下一步修改的知识`
- `马上回项目里验证`

这样你补出来的，不只是文档和概念，而是真正能变成下一轮代码收口的能力。
