# 其他算子、算子融合与优化顺序依赖说明

## 1. 这份文档回答什么问题

这份文档回答三个问题：

1. 是不是只有矩阵乘法才适合“分层合同 + 分步优化”这种模式
2. 其他算子、算子融合、attention 之类的优化，是否也是类似模式
3. 优化 pass 的顺序能不能调换，哪些必须先做

本文基于当前仓库真实 stage1 主线，而不是抽象编译器教材。

## 2. 先说结论

结论很明确：

1. 不只是矩阵乘法。大多数高性能算子优化，都会遵循“先固定真相，再逐层优化”的模式。
2. 但不同算子的中间专项合同不一样，不会和 matmul 完全同构。
3. pass 顺序不是完全不能动，但有一条很强的硬依赖链：前一层不成立，后一层通常就没有可靠输入。
4. 越靠后的 lowering pass，越不应该重新猜上层语义；它应该主要消费前面已经公开的合同。

换句话说，真正稳定的优化体系，不是“某个 pass 很聪明”，而是“每一步都知道自己消费什么、拥有何种真相、给下一步交出什么”。

---

## 3. 通用模式：大多数高性能优化都长这样

无论是 matmul、reduction、softmax、attention，还是算子融合，通常都可以压成下面这条通用链：

```text
高层语义
  -> 合法性 / 目标硬件信息
  -> 并行映射
  -> 布局 / 编码 / 数据流组织
  -> 资源与生命周期
  -> 调度 / 流水 / 同步
  -> target lowering
```

这条链背后的含义是：

1. 先回答“我到底在算什么”
2. 再回答“在当前硬件上哪些实现方式合法”
3. 再回答“谁负责哪块数据”
4. 再回答“数据在 global/shared/register 里怎么组织”
5. 再回答“什么时候搬、什么时候算、什么时候等”
6. 最后才把这些真相翻译成目标后端 IR / PTX / SASS

如果顺序反过来，比如还没讲清楚 layout 和 transport，就先做 async 或 lowering，最后通常会变成：

- 后端在猜
- pass 边界不干净
- 结构很难扩展
- 一改形状或一加 fallback 就碎

## 4. 当前项目的真实顺序

当前仓库 stage1 主线顺序在：

- `/home/zhangruiqi/mini_triton_nvgpu_v1/lib/Transforms/Stage1Pipelines.cpp`

主线大致是：

```text
tb.matmul
  -> verify-scope
  -> attach-target-info
  -> semanticize-matmul
  -> build-program-mapping-plan
  -> build-layout-plan
  -> build-transport-plan
  -> build-c-register-plan
  -> rewrite-matmul-mainloop
  -> build-mainloop-graph
  -> regularize-k-loop
  -> assign-latencies
  -> schedule-loops
  -> derive-waits
  -> expand-pipeline
  -> cleanup-pipeline
  -> verify-pipeline-mainline-contract
  -> lower-pipeline-to-nvgpu
  -> lower-epilogue-vector-io
  -> NVGPU / NVVM / PTX
```

这条链不是随便排的，而是对应了一条真实的依赖关系：

- 不先知道 scope / target，就不能知道 `mma.sync` 和 `cp.async` 合法不合法
- 不先知道语义和 program mapping，就不能做稳定的 layout
- 不先知道 layout / transport，就不能做 async producer / consumer wait
- 不先建 buffer/resource graph，就不能做 schedule 和 wait 推导
- 不先把 pipeline mainline 显式化，后端 lowering 就会偷偷重建一套逻辑

## 5. 不同算子是否也是这个模式

答案是：大方向上是，但专项合同不同。

### 5.1 矩阵乘法

matmul 是最典型、也最完整的一条链。

它几乎要求把下面这些层都做出来：

1. 语义层
2. CTA/program 映射层
3. layout/encoding 层
4. transport 层
5. accumulator/epilogue 层
6. buffer/resource graph 层
7. latency/schedule/wait 层
8. pipeline mainline 层
9. target lowering 层

原因是 matmul 的性能高度依赖：

- fragment 组织
- shared/register 复用
- global 到 shared 到 register 的多级搬运
- copy/compute overlap
- epilogue 路径

所以它特别适合用“分层 owner truth”的方式做。

### 5.2 Reduction

reduction 的总体模式和 matmul 很像，但专项真相不同。

共通部分：

- 语义
- 并行映射
- 资源图
- 调度
- lowering

不同部分：

- matmul 关注 dot operand、accumulator、mma fragment
- reduction 关注 reduction 轴、partial result owner、warp/CTA combine 路径、shared tree 或 shuffle 归约

如果以后给 reduction 建专项合同，通常会需要这类结构：

- `ReductionSemantics`
- `ReductionMappingPlan`
- `ReductionTreePlan`
- `ReductionBufferPlan`
- `ReductionSyncPlan`

所以 reduction 也不是“直接写一个 kernel 就完了”，它同样需要先固定中间真相。

### 5.3 算子融合

fusion 也遵循同样思想，但关键步骤更靠前。

fusion 的核心不是 `mma.sync`，而是：

1. 哪些 op 可以合法融合
2. 融合后 region 的边界是什么
3. producer/consumer 中间值能不能消掉
4. indexing / shape / broadcast 是否兼容
5. 融合后哪个算子是主体

因此 fusion 通常在高层语义期先做，而不是等到底层 fragment 或 NVGPU 阶段再做。

典型逻辑是：

```text
高层 op
  -> fusion legality
  -> fusion region construction
  -> fused semantics
  -> 再进入 layout / mapping / transport / lowering
```

这也意味着：

- fusion 常常是“更前面的 pass”
- fusion 完成后，融合后的主体 region 再进入 matmul/reduction 等专项优化主线

### 5.4 Attention

attention 更像是 matmul、reduction、fusion 的组合体。

它往往需要同时固定：

- Q/K/V 的布局和语义
- block mapping
- score compute
- softmax state
- online reduction state
- V accumulation
- epilogue/store path
- persistent work / tile reuse
- pipeline overlap

因此 attention 比 matmul 更需要分层合同，而不是更少。

## 6. 哪些层是共通骨架，哪些层是专项合同

### 6.1 共通骨架

几乎所有高性能算子都离不开下面这些层：

1. 语义层
2. 合法性 / 目标硬件层
3. 并行映射层
4. 资源 / 生命周期层
5. 调度 / 同步层
6. target lowering 层

这些层是“通用骨架”。

### 6.2 专项合同

不同算子真正拉开差异的，是中间专项合同。

例如：

- matmul：dot operand encoding、accumulator、mma mainloop、async global->shared transport
- reduction：reduction tree、partial result owner、cross-thread combine
- fusion：fusion legality、fusion region、producer-consumer inline 边界
- attention：softmax state、online reduction、persistent work、K/V reuse

也就是说：

- 骨架是通用的
- 中间合同是专项的

## 7. 优化顺序是不是不能调换

答案分成两部分。

### 7.1 有些顺序基本不能乱

这些属于硬依赖顺序。

#### 1. `verify-scope` / `attach-target-info`

必须早。

原因：

- 不知道目标架构和硬件能力，就没法判断 `cp.async` / `ldmatrix` / `mma.sync` 合法性
- 不知道当前 stage 支持范围，后面的优化可能建立在错误前提上

#### 2. `semanticize`

必须早。

原因：

- 不先知道“这个 op 的真实数学语义和 tile 语义是什么”，后面的 mapping/layout 都没有稳定输入

#### 3. `program mapping`

通常必须早于 layout/pipeline。

原因：

- 不先决定 CTA / program 负责哪块 tile，后续 CTA 内 layout 和调度就会和 CTA 外部映射混在一起

#### 4. `layout / encoding`

通常必须先于 transport / async / mma lowering。

原因：

- 不先知道 fragment、shared layout、lane ownership，就不知道 operand 怎样合法进入 `ldmatrix` / `mma.sync`

#### 5. `transport`

通常必须先于 async wait / schedule。

原因：

- 不先知道 global->shared 的 vector bytes、async legality、cache policy，就不知道 async producer 是什么

#### 6. `resource graph / buffer model`

通常必须先于 latency / schedule / waits。

原因：

- schedule 和 waits 需要基于谁生产、谁消费、谁复用、谁覆盖来推导
- 没有资源图，只能对 IR 表面形状做脆弱推理

#### 7. `loop regularization`

通常必须先于真正的 pipeline pass。

原因：

- 不先把 loop owner 规整出来，schedule/pipeline 往往会绑死在偶然的 IR 形状上

#### 8. `latency -> schedule -> waits`

这几个一般有明显的先后依赖。

原因：

- 先有 latency 建模，才好做时间排布
- 有了 schedule，才知道 wait 应该放在谁前面
- 没有 schedule，wait placement 很容易靠猜

#### 9. `pipeline expansion / cleanup / verify`

必须在 lowering 前。

原因：

- lowering 不应该自己发明 pipeline
- 它应该只消费显式的 `tb.pipeline_mainline`

### 7.2 有些顺序可以调，但不能破坏语义依赖

这类属于实现型顺序。

例如：

- 某些 cleanup/canonicalize 插在哪一段
- 某些 analysis 是拆成两个 pass 还是合成一个 pass
- latency 和 schedule 是否做迭代式细化
- fusion legality 和 fusion region 是拆开还是合并

这些地方可以调，但前提是：

- 不能破坏 owner truth
- 不能把后一层该消费的合同重新藏回实现细节

所以更准确的说法不是“顺序绝对不能改”，而是：

**语义依赖链不能乱，实现落点可以调整。**

## 8. 一个很典型的例子：为什么 `derive-waits` 不能提前

`derive-waits` 不是凭空做的，它依赖很多前置真相：

1. producer 是谁
2. producer 属于哪个 async group
3. consumer 第一次什么时候用
4. 当前 schedule 怎么排
5. shared slot 什么时候会被覆盖

这些信息分别来自：

- `TransportPlan`
- `BufferModel`
- `LatencyPlan`
- `PipelinePlan`

所以如果把 wait pass 提到前面，它拿不到稳定输入，只能猜。

这就是“为什么有些 pass 必须先做前置层”的最直接例子。

---

## 9. 一个同样典型的例子：为什么 fusion 往往更靠前

fusion 通常在高层做，而不是在低层做。

原因是 fusion 关心的是：

- 依赖关系
- side effect
- shape / broadcast 兼容性
- producer / consumer region 是否能并

这些都是高层语义问题。

如果已经 lower 到：

- fragment
- shared staging
- async wait
- mma cluster

这种层次，再做融合通常就太晚了，成本高，而且很难保持结构干净。

因此：

- fusion 往往是高层优化
- pipelining / async / target lowering 往往是更低层的执行优化

## 10. 对当前项目的启发

你当前项目的 stage1 之所以能一步步优化，是因为它已经遵循了这条通用原则：

1. 先固定语义
2. 再固定映射
3. 再固定 layout/transport
4. 再固定资源图和时序图
5. 最后才 lowering

这也是为什么当前项目特别适合继续往下扩展：

- 如果以后做 reduction，可以复用骨架，补 reduction 专项合同
- 如果以后做 fusion，可以在高层插入 fusion legality / region pass
- 如果以后做 attention，也还是这个模式，只是专项合同更多

因此你现在这个项目不是“只适合 matmul 的一次性写法”，而是已经具备了走向更广泛高性能优化框架的基本形态。

---

## 11. 最后用一句话收口

高性能优化的大规律不是“先写快代码”，而是：

**先把算子语义、并行分工、布局、搬运、资源、同步这些真相一层层固定成显式合同，再让后面的 pass 去消费这些合同。**

所以：

- 其他算子大体也是这个模式
- fusion 也一样，只是更靠前
- pass 顺序不是完全僵死，但硬依赖链不能乱
- 真正成熟的后端，优化顺序看起来像很多 pass，本质上是在逐层减少不确定性
