- `GPU 编译器 / MLIR / CUDA kernel / AI compiler / 性能分析` 相关实习或工作准备

## 一、先说结论

真实工作通常确实像现在这个项目一样复杂。

但这种复杂不是指：

- 每个人都要同时把所有层做到最深

而是指：

- 问题链条经常跨很多层
- 你至少要知道每层在做什么
- 你要能沿着链条定位问题、归因问题、验证修改

更准确地说，这个方向要求的是：

- 一块主轴能力足够深
- 其余相关层能看懂、能协作、能定位

这是一种典型的 `T 型能力结构`。

---

## 二、真实工作为什么会复杂

## 1. 复杂在“跨层”，不只在“代码多”

一个 matmul 性能问题，现实里往往不是单层问题，而是可能同时涉及：

- 模型侧 shape 分布和热点算子
- 框架或 graph compiler 的算子选择
- 编译器 pass 的 layout / mapping / pipeline 决策
- target lowering 发出的 PTX / SASS 形态
- runtime 的 launch / cache / sync 开销
- 硬件上的 occupancy、stall、memory 行为

所以真实复杂度主要来自：

- 你需要顺着链条追问题
- 你不能只盯着某一个文件或某一个 kernel

## 2. 复杂在“边界”

真实工作里最常见的问题不是“这个 API 会不会用”，而是：

- 这个真相到底该放在哪层
- 这个 bug 是 framework 选错了，还是 compiler 改坏了
- 这个慢是因为 runtime 口径不公平，还是 kernel 本身真的慢
- 这个优化应该放 pass 里，还是放 lowering 里，还是交给 autotune

也就是说，很多工作其实是在做：

- 职责边界划分
- owner truth 收口
- contract 提前验证

## 3. 复杂在“证据链”

真实工作通常不能只说“我感觉这里慢”。

你需要把下面这些串起来：

- benchmark 结果
- IR 形态
- PTX / SASS
- `ncu / nsys` 指标
- 资源占用
- 修改前后对比

最后才能比较有把握地说：

- 根因是什么
- 该改哪层
- 为什么这个修改是对的

## 4. 复杂在“工程成本”

真实工作不是只做一个最快的 case，而是要考虑：

- 这个修改会不会把 fallback 改坏
- 会不会让 general-shape 退化
- 会不会让测试覆盖断掉
- 会不会让后续维护成本变高

所以真正的复杂，不只是算法复杂，而是：

- 性能
- 正确性
- 覆盖范围
- 工程可维护性

这四件事要一起平衡。

## 三、真实工作里常见的几类日常任务

不同团队主轴不同，但日常大致会落在这些工作上。

## 1. 编译器主线开发

典型内容：

- 设计 attr / type / op / contract
- 写 analysis / rewrite / cleanup / verifier / lowering pass
- 调整 pass pipeline 顺序
- 让某类 op 更稳定地命中优化路径

## 2. kernel 与 codegen 优化

典型内容：

- 调整 layout / fragment / pipeline
- 改 async copy / shared layout / epilogue path
- 检查 PTX / SASS 是否符合预期
- 把上层合同正确翻译到 target IR

## 3. profiling 与性能归因

典型内容：

- 跑 benchmark
- 跑 `nsys / ncu`
- 分析 occupancy、stall、memory throughput、tensor pipe 利用率
- 对比修改前后结果

## 4. correctness / regression / infra

典型内容：

- 写功能测试
- 写 lit / FileCheck
- 搭 benchmark harness
- 做性能回归检查

## 5. 跨团队对齐

典型内容：

- 和模型同学确认真实热点 shape
- 和框架同学确认 graph / layout / op selection
- 和 runtime 同学确认 launch / caching / scheduling 行为
- 和硬件或性能同学一起看 profiler 结果

所以真实工作通常不是只写 pass，也不是只看 profiler，而是：

- 以某一层为主
- 但会跨几层来回工作

## 四、一个 matmul 性能问题在真实工作里通常怎么排查

下面这条链路，是最值得你现在建立的真实工作思维。

## 第 1 步：先确认问题是不是“真问题”

先看：

- 测量口径是否公平
- 是否已经预热
- 是否比较的是同一 shape、同一 dtype、同一 launch 配置
- 是否把 runtime 初始化、JIT、cache miss 等因素混进去了

这一层如果没确认，后面所有分析都可能偏掉。

常见误区：

- 单轮跑分就下结论
- 比较双方 shape 不一致
- 一边热机、一边冷机
- 把 launch overhead 当成 kernel 差距

## 第 2 步：确认真实 workload 和命中路径

先问：

- 真实模型里慢的是哪类 shape
- 当前 shape 是否 exact-tile
- 当前是否命中了预期 kernel / pass 主线
- 有没有退到 fallback

这一层常常需要看：

- 输入 shape
- launch config
- pass 输出
- 是否命中 direct path / fallback path

## 第 3 步：看 compiler IR 有没有走对主线

这里要看：

- `ProgramMappingPlan`
- `EncodingPlan`
- `TransportPlan`
- `EpiloguePlan`
- `PipelinePlan / AsyncPlan / PipelineExpansion`
- `tb.pipeline_mainline`

核心问题是：

- 上层合同是不是已经决定成了你想要的那条主线
- 还是说一开始 owner 就选错了

如果这一步错了，后面看 PTX / SASS 只能看到“坏结果”，但看不到最早根因。

## 第 4 步：看 lowering、PTX、SASS 是否忠实消费了上层合同

这里要确认：

- lowering 有没有偷偷重建主线
- 有没有在 target lowering 里再次猜 layout / mapping / landing kind
- PTX / SASS 是否出现预期指令
- 有没有非预期的 scalarization、额外 barrier、错误的 shared relay

这一层的重点不是“盯着汇编看”，而是：

- 检查 lower 出来的东西是否和上层 contract 一致

## 第 5 步：看 runtime 行为

这里常见问题是：

- launch overhead
- stream / sync 使用方式
- workspace 或缓存行为
- JIT 或 binary cache

很多时候 kernel 本身没那么差，但系统侧行为把收益吃掉了。

## 第 6 步：看硬件指标

这一层常用工具是：

- `Nsight Systems`
- `Nsight Compute`

常看问题包括：

- occupancy 是否过低
- register pressure 是否过高
- static / dynamic shared 是否把资源压满
- memory throughput 是否异常
- tensor pipe 利用率是否不足
- shared bank conflict 是否明显
- stall 主要来自哪里

这一层帮助你判断：

- 这是算法/合同问题
- 还是资源与硬件行为问题

## 第 7 步：把根因归类

一个性能问题，最后通常会落到下面几类之一：

### A. 上层 owner 选错

例如：

- 该 direct 的走成 relay
- 该 async 的退成同步路径
- mapping 一开始就不对

### B. lowering 没有忠实翻译合同

例如：

- 上层 plan 对了，但 lower 又私自重建
- shared 物理访问没按 encoding owner 走
- epilogue path 在 target lowering 被改歪

### C. 资源 tradeoff 不对

例如：

- register 太高
- shared 太大
- occupancy 掉太多
- 某条路径虽然指令少，但资源更差

### D. 测量口径问题

例如：

- benchmark 不公平
- 热机不一致
- launch 配置不一致

只有先归到这一类，你后面的修改才不会乱。

## 第 8 步：决定该改哪层

这是最关键的一步。

你要先问：

1. 这个根因是谁的 owner？
2. 这件事应该在哪一层决定？
3. 我现在是在修根因，还是只是在 lower 末端打补丁？

理想顺序通常是：

- 先改 owner 层
- 再改 cleanup / verifier
- 最后改 lowering 消费逻辑

而不是反过来。

## 第 9 步：做修改后重新验收

至少要重新看：

- correctness
- IR 是否符合预期
- PTX / SASS 是否符合预期
- `ncu / nsys` 是否支持你的归因
- benchmark 是否真的改善
- 是否把其他 shape / fallback 改坏

这一步决定你是“试出来了一个结果”，还是“完成了一次可信的性能工程修改”。

## 五、把这条真实工作链路映射回你当前项目

你当前项目已经能很好地练到这条链路里的核心部分。

## 1. 问题和目标层

对应：

- `KernelConfig`
- `TargetInfo`
- `MatmulSemantics`

你在这里练的是：

- 问题规模、tile、硬件能力、exact-tile 语义

## 2. 上层 owner 决策层

对应：

- `ProgramMappingPlan`
- `EncodingPlan`
- `TransportPlan`
- `AccumulatorPlan`
- `EpiloguePlan`

你在这里练的是：

- mapping、layout、transport、C landing 的 owner truth

## 3. 主线执行组织层

对应：

- `BufferModel`
- `LoopPlan`
- `LatencyPlan`
- `PipelinePlan`
- `AsyncPlan`
- `PipelineExpansion`
- `tb.pipeline_mainline`

你在这里练的是：

- pipeline、async、wait、cluster、执行主线

## 4. target lowering 层

对应：

- `LowerPipelineToNVGPU.cpp`
- `LowerEpilogueVectorIO.cpp`

你在这里练的是：

- 如何把上层合同忠实落到 target IR

## 5. benchmark / profiling / 归因层

对应：

- `tools/bench_stage1_fair.py`
- 你现在做的 `ncu / nsys / PTX / SASS` 对比

你在这里练的是：

- 真实性能工程的证据链

所以你现在这个项目，已经不是只在练“写一个 kernel”，而是在练：

- 编译器边界
- 性能边界
- 工程边界

## 六、从实习准备角度，真正要练成什么

如果你准备的是相关实习，不要求你现在就能独立撑起整条工业级栈。

但你至少要练成下面这些能力：

## 1. 能讲清一条主线

你要能讲清楚：

- 你的项目主线是什么
- 每一层谁是 owner
- lowering 为什么不该回头猜

## 2. 能解释一个性能问题

你要能从：

- shape
- IR
- PTX / SASS
- `ncu / nsys`
- benchmark

串成一条完整解释链。

## 3. 能指出当前边界

你要能诚实地说清楚：

- 现在 exact-tile 很强
- 但 general-shape、shared-pack、split-k/persistent 还没完全收口

这不是减分，反而会让你显得成熟。

## 4. 能提出正确修改方向

你要能说：

- 这个问题该改哪层
- 为什么不是在 lower 末端补一层 if/else
- 为什么要先补 contract、再补 verifier、最后补 lowering

---

## 七、一句话总结

真实工作确实复杂，但这种复杂不是“每个人都必须同时精通所有层”，而是：

- 你要有一条主轴能力
- 还要能跨层看问题、定位问题、归因问题

对你现在最重要的，不是“把所有层都学完”，而是：

- 先把一条真实链路学会
- 能讲清每层负责什么
- 能把一个性能问题从 shape 一路追到 IR、PTX、硬件指标，再一路归因回来

这就是这个方向最核心、也最接近真实工作的能力。
