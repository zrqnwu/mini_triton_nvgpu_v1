## 一、先说结论

你现在不需要先把：

- LLVM backend 全学完
- Triton 全源码全啃完
- PTX ISA 全背完

再来读这个项目。

你现在真正需要的是：

1. 先掌握这个项目依赖的最小基础知识
2. 按正确顺序读它，而不是从最大的 lowering 文件开始
3. 一边读一边回答：
   - 这层负责什么
   - 为什么要放在这一层
   - 为什么不能拖到下一层再猜

只要你按这个方法读，这个项目虽然复杂，但并不会乱。

## 二、读这个项目之前，你至少要知道哪些基础知识

## 1. MLIR 基础

你现在至少要知道下面这些概念：

- `op` 是什么
- `attr` 是什么
- `type` 是什么
- `pass` 是什么
- pass 大致分成：
  - verify
  - analysis/build-contract
  - rewrite
  - cleanup/canonicalize
  - lowering

你不需要一开始就精通：

- dialect conversion 全部细节
- region/interface/trait 全量体系
- MLIR 所有官方 dialect

对这个项目来说，你最需要先理解的是：

- 哪些真相应该放 `attr/type`
- 哪些真相应该放显式 `op`
- 哪些真相应该由 pass 构造
- lowering 为什么最好只做翻译

对应代码入口：

- [TBOps.td](/home/zhangruiqi/mini_triton_nvgpu_v1/include/tb/IR/TBOps.td)
- [Passes.td](/home/zhangruiqi/mini_triton_nvgpu_v1/include/tb/Transforms/Passes.td)

## 2. CUDA / GPU 基础

你至少要知道：

- `thread / warp / CTA(block)`
- `global / shared / register`
- `occupancy`
- `register pressure`
- `shared memory pressure`
- 为什么 warp 内访问模式会影响性能

你不需要一开始就会：

- 所有 PTX 指令
- 所有新架构特性
- 全部 CUTLASS / CuTe 细节

对这个项目来说，最低要能看懂：

- `cp.async` 大致是干什么的
- `ldmatrix` 为什么需要特定 shared 布局
- `mma.sync m16n8k16` 是 Tensor Core 主线的一部分

## 3. 矩阵乘法分层基础

你至少要知道：

- problem shape
- block tile / CTA tile
- warp tile
- instruction tile

不然你会看不懂：

- `ProgramMappingPlan`
- `EncodingPlan`
- `AccumulatorPlan`
- `MatmulRewritePlan`

## 4. 性能分析基础

你至少要知道：

- benchmark 为什么要公平
- 单轮跑分为什么不可靠
- `ncu / nsys` 是什么
- runtime、IR、PTX、SASS、硬件指标为什么要串起来看

这不是“后面再说”的内容，因为这个项目本身就不是只追求“能 lower”，而是追求：

- 正确
- 可解释
- 有性能闭环

## 5. 编译器项目阅读方法

这是最重要的基础。

你要先接受一个事实：

- 不能从最大的 `.cpp` 文件开始读
- 也不能按文件夹顺序乱翻

正确方法是：

- 先看总 pipeline
- 再看 op 和 pass 声明
- 再看每层 analysis contract 的头文件

## 五、这个项目为什么要这样设计

## 1. 因为它不想把所有真相都堆进 lowering

这是这个项目最重要的设计动机。

如果所有决策都拖到 lowering 才做，会出现：

- layout 在 lowering 猜一次
- mapping 在 lowering 猜一次
- epilogue landing 在 lowering 猜一次
- async/wait frontier 在 lowering 猜一次

那最后项目会变成：

- 一个非常大的 lowering 文件
- 很多难以解释的隐式决策
- 很难做 verifier
- 很难做结构性重构

所以它故意把真相往前提：

- 用 attr/type/op 显式表达
- 用 analysis contract 提前收口
- 用 cleanup/verifier 在 lowering 前封口

## 2. 因为 layout、transport、epilogue、pipeline 本来就是不同问题

它们都和性能有关，但不应该混成一团。

- layout
  - 值怎么摆
- transport
  - 值怎么搬
- epilogue
  - C 怎么落地
- pipeline
  - 主循环怎么 overlap、怎么等

这就是为什么会有：

- `EncodingPlan`
- `TransportPlan`
- `EpiloguePlan`
- `PipelinePlan / AsyncPlan / PipelineExpansion`

## 3. 因为 `tb.pipeline_mainline` 是执行主线的显式 owner

这是项目最值钱的点之一。

[tb.pipeline_mainline](/home/zhangruiqi/mini_triton_nvgpu_v1/include/tb/IR/TBOps.td#L60) 的意义是：

- cleanup 之后，执行主线已经被显式公开
- target lowering 不应该再偷偷重建 skeleton

也就是说：

- 高层合同已经收口完成
- lowering 只负责消费合同并翻译

## 4. 因为 verifier 和 lowering 应该分开

这也是一个很重要的工程判断。

[tb-verify-pipeline-mainline-contract](/home/zhangruiqi/mini_triton_nvgpu_v1/include/tb/Transforms/Passes.td#L184) 的存在说明：

- 合同完整性最好在 lowering 前检查
- 不应该把所有“坏合同”都留到 target lowering 晚期才报错

这样做的好处是：

- 报错更早
- 错误更清楚
- lowering 更干净

## 六、为什么顺序必须是现在这样

这一条非常重要。

当前顺序不是“作者随手排的”，而是严格按依赖链来的。

## 1. 先 verify scope

为什么：

- 先确认当前 `stage1` 合法范围
- 避免后面所有分析都建立在非法 case 上

## 2. 再 attach target info

为什么：

- 后面 layout、transport、lowering 都依赖硬件能力
- 这些信息必须先成为显式 owner

## 3. 再 semanticize

为什么：

- 先把“数学语义 + tile 语义 + 边界语义”讲清楚
- 物理布局和执行形态不能在这之前决定

## 4. 再 program mapping

为什么：

- 先定“哪个 CTA 干哪块活”
- 才能谈 CTA 内部 layout 和 pipeline

## 5. 再 layout

为什么：

- 先定物理 encoding
- 后面 transport、fragment、`ldmatrix` 才有基础

## 6. 再 transport

为什么：

- transport 是建立在 layout 合法性上的
- 不是先拍脑袋决定 async/vector width，再去适配 layout

## 7. 再 accumulator / epilogue

为什么：

- C 路径必须在 rewrite 前已经有合同
- 不能等到 lowering 才想“这次 direct 还是 relay”

## 8. 再 matmul rewrite

为什么：

- 这一步才把前面的语义/布局/C 合同收成真正的 tensor-core mainloop 形态

## 9. 再 buffer model / loop / latency / schedule / waits / expansion

为什么：

- 这是从“合同”走向“执行主线”的必经过程
- 先得有统一资源图，后面才能做 loop、latency、schedule、async

## 10. 再 cleanup 和 verifier

为什么：

- cleanup 把执行主线封成显式 `tb.pipeline_mainline`
- verifier 再确认 lowering 合同已经完整

## 11. 最后才 lowering

为什么：

- 到这个阶段，应该已经不存在“再回头做上层决策”的空间
- lowering 只做翻译，不再做大规模 owner 选择

## 十、一句话总结

如果你现在只有基础 `MLIR` 和基础 `CUDA`，读这个项目最重要的不是“先把所有知识补全”，而是：

- 先顺着真实 pass 主线看
- 先搞清每层谁是 owner
- 先理解为什么这个顺序不能乱

只要你沿着：

- `pipeline 总览`
- `IR 边界`
- `pass 边界`
- `analysis contract`
- `pipeline_mainline`
- `lowering`

这条线去读，这个项目就会从“很复杂的一堆代码”变成“结构非常清楚的一条编译器主线”。
