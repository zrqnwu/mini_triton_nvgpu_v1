## 3. 这轮真正修掉的东西

### 3.1 修掉的根因

这轮修掉的是 `Stage1 / Step8` 里一条错层依赖：

- 旧逻辑把 `program.waits` 里的 `load->mma` 可见性，直接投成 `device_async_wait`
- 但 executor 里真正的 shared consumer 是 `ldmatrix`
- 所以旧生成顺序变成：
  - 先 `ldmatrix`
  - 后 `device_async_wait`

这会直接导致 exact-tile correctness 出错，之前一度出现“结果只剩一半”的坏状态。

### 3.2 现在的正确做法

这轮在下面文件里删掉了旧逻辑并重写：

- `/home/zhangruiqi/triton_backend_nvgpu/lib/tb/AsyncCopyPlan.cpp`

新的逻辑是：

- `device_async_wait` 不再吃 `program.waits` 的 `load->mma` frontier
- 它现在直接锚到同一 `kGroup` 的第一条真实 shared consumer：
  - `load_a`
  - `load_b`

对应删除原因已经写到：

- `/home/zhangruiqi/docs/triton_backend_nvgpu_stage1_round3_deletion_rationale.md`

重点位置：

- `2.15 删除：把 program.waits 的 load->mma 可见性直接当成 async wait 真相`

### 3.3 这轮修复为什么可信

不是只看输出对了，而是中间 IR 顺序也对了。

坏版本：

- `device_async_wait` 在第一批 `ldmatrix` 之后

新版本：

- `/tmp/tb_stage1_perf_20260330_runtime_new_raw.mlir:5068`
  - `nvgpu.device_async_wait`
- `/tmp/tb_stage1_perf_20260330_runtime_new_raw.mlir:5083`
  - 第一条 `nvgpu.ldmatrix`

也就是说，当前已经实现成：

- 先 wait
- 再第一条真实 shared 读

这才是 Step8 当前这层该承担的正确语义。
